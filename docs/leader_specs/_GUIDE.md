# リーダーカード効果 テスト仕様書 — 作成ガイド

## 目的
全137リーダーカードを **1枚ずつテキスト精読** し、効果の **意味的な正しさ** を検証する
テスト仕様書を作る。既存の自動ツール（`full_card_audit` の構造監査＝クラッシュ/カード消失、
`effect_coverage` の方向監査）では拾えない **意味バグ** を発見することが狙い。

意味バグの例（既存ツールが見逃すもの）:
- 対象が逆（自分↔相手）、対象範囲の条件欠落（コスト/パワー/特徴/枚数）
- 数値違い（+2000のはずが+1000 等）、対象プレイヤー違い
- **条件節の欠落**（例「Aまたは8枚以上」の "または8枚以上" が落ちている）
- 効果の一部が未実装で `NO_CHANGE`、本来あるべき効果が起きない
- 「までを」(is_up_to) / ターン1回 / 継続期間(THIS_TURN等) の取りこぼし
- 「持ち主の」「元々のコスト」等の細部の取りこぼし

## 入力（プローブ）
各セットの全リーダーについて、テキスト＋パース結果(AST要約)＋実行観測を出力:

```
OPCG_LOG_SILENT=1 python tests/leader_spec_probe.py --set OP01 2>&1 | grep -v "GCS Client"
# 単体: ... leader_spec_probe.py OP01-001
```

出力の見方:
- **効果テキスト**: 原文（これが正）
- **パース結果**: `summarize_ability` 指紋（エンジンがどう解釈したか）
- **実行観測 (classify)**: 汎用盤面で能力を発動した結果の status と盤面差分
  - `EXECUTED` 盤面が変化 / `NO_CHANGE` 変化なし(条件未達 or 未実装) /
    `INTERACTIVE` 選択待ちで停止 / `ERROR` 例外

## 各リーダーの記述テンプレート

```markdown
## <ID> <名前>
- **基本**: 色=… / ライフ=… / パワー=… / 属性=… / 特徴=…
- **効果テキスト**:
  > <原文をそのまま>
- **能力分解（テキスト解釈）**: 能力ごとに箇条書き
  - 能力0 【トリガー】: コスト=… / 効果=… / 対象=… / 数値=… / 期間=… / 条件=… / ターン1回=…
- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |
  |---|----------|------|----------|
  | 1 | … | 能力発動 | … |
  - 境界/異常系: 対象不在 / 条件未達 / コスト払えない 等
- **パース照合**: パース結果がテキスト通りか。食い違いを具体的に。
- **実観測照合**: classify の status/差分が期待通りか。食い違いを具体的に。
- **判定**: ✅問題なし / ⚠️要確認 / 🐛バグ — <根拠を一文で>
```

## 判定ラベル
- ✅ **問題なし**: パース・実観測ともテキストと整合（手動確認した範囲で）
- ⚠️ **要確認**: 断定できないが疑わしい / 汎用盤面では検証不能で要個別テスト
- 🐛 **バグ**: テキストと明確に食い違う（対象・数値・条件欠落・未実装など）
