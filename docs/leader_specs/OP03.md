## OP03-001 ポートガス・D・エース
- **基本**: 色=赤 / ライフ=5 / パワー=5000 / 属性=特 / 特徴=白ひげ海賊団
- **効果テキスト**:
  > このリーダーがアタックした時かアタックされた時、自分の手札から任意の枚数イベントかステージカードを捨ててもよい。捨てたカード1枚につき、このリーダーは、このバトル中、パワー+1000。
- **能力分解（テキスト解釈）**:
  - 能力0 【アタックした時／アタックされた時】: コスト=なし / 効果=手札からイベントorステージを任意枚数捨てる(任意=してもよい) → 捨てた枚数×1000、このバトル中このリーダーにパワー+ / 対象=自分の手札(EVENT,STAGE) / 数値=捨て枚数×1000 / 期間=THIS_BATTLE / 条件=なし / ターン1回=なし
- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |
  |---|----------|------|----------|
  | 1 | 手札にイベント2枚 | アタック時に2枚捨て | リーダーパワー+2000(このバトル) |
  | 2 | 手札に捨てる対象なし | アタック時 | 何も捨てず+0、能力スキップ可 |
  | 3 | アタックされた時 | 同上トリガー発火 | 捨て枚数×1000 |
  - 境界/異常系: イベント/ステージ以外(キャラ)は捨て対象外 / 0枚選択も可(任意)
- **パース照合**: トリガーは`ON_ATTACK`のみ。テキストは「アタックした時**か**アタックされた時」だが、被アタック側トリガー(ON_ATTACKED/ON_BLOCK相当)が指紋に現れない。`ON_ATTACK`単独だと「アタックされた時」が欠落の疑い。DISCARD対象card_typeは[LEADER,EVENT,STAGE]でLEADERが余分(無害だが手札にLEADERは通常無い)。count=50 is_up_true=true / is_optional=true / 動的値PREV_ACTION_COUNT×1000は妥当。
- **実観測照合**: `ON_ATTACK: INTERACTIVE`（捨てる枚数の選択待ち）。妥当。被アタック側は未検証。
- **判定**: ⚠️ 要確認 — 「アタックされた時」トリガーが`ON_ATTACK`に含まれるか不明、個別テスト要。

- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |

## OP03-040 ナミ
- **基本**: 色=青 / ライフ=5 / パワー=5000 / 属性=知 / 特徴=東の海
- **効果テキスト**:
  > ルール上、自分のデッキが0枚になった場合、自分は敗北する代わりに勝利する。 / 【ドン!!×1】このリーダーのアタックによって、相手のライフにダメージを与えた時、自分のデッキの上から1枚をトラッシュに置いてもよい。
- **能力分解（テキスト解釈）**:
  - 能力0 【常時/ルール】: デッキ0枚で敗北の代わりに勝利(REPLACE_DECKOUT_LOSS) / 条件=自デッキ==0
  - 能力1 【ドン!!×1】【ダメージを与えた時】: このリーダーのアタックで相手ライフにダメージ → 自デッキトップ1枚をトラッシュに置いてもよい(任意) / 条件=ドン!!1付与 / トリガー=ライフダメージ与時
- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |
  |---|----------|------|----------|
  | 1 | 自デッキ0枚 | 状態判定 | 敗北でなく勝利 |
  | 2 | ドン1付与・アタックがライフ貫通 | ダメージ時 | デッキトップ1枚トラッシュ(任意) |
  | 3 | アタックがブロック/ライフ未到達 | — | 能力不発 |
  - 境界/異常系: デッキ0でないと勝利化しない / ライフにダメージ無しなら不発
- **パース照合**: 能力1のトリガーが`ACTIVATE_MAIN`になっている。**テキストは「相手のライフにダメージを与えた時」＝アタック起因の誘発トリガー**(ON_DAMAGE/ライフダメージ時相当)であり、起動メインではない。**トリガー種別が誤り**の疑い。effectはTRASH_FROM_DECK(value=1,is_optional)で内容は妥当。能力0(VICTORY,REPLACE_DECKOUT_LOSS)は整合。
- **実観測照合**: PASSIVE/ACTIVATE_MAIN とも`NO_CHANGE`。能力0はデッキ0でないため不発(妥当)。能力1はACTIVATE_MAIN扱いだが誘発条件(ライフダメージ)が無く不発。
- **判定**: 🐛 バグ — 能力1のトリガーが`ACTIVATE_MAIN`化しており、本来の「相手ライフにダメージを与えた時」誘発になっていない疑い。

  |---|----------|------|----------|
  | 1 | — | アタック宣言 | アタック不可 |
  | 2 | ドン!!1返却可・リーダーactive・手札にコスト5《GC》 | 発動 | リーダーrest、《GC》キャラ登場 |

- **判定**: ⚠️ 要確認 — コスト処理は動作するがPLAY_CARDの方向不一致WARNが出る、対象《GC》キャラを置いた個別テストで登場が正しく起きるか確認要。

## OP03-077 シャーロット・リンリン
- **基本**: 色=黒/黄 / ライフ=4 / パワー=5000 / 属性=特 / 特徴=四皇・ビッグ・マム海賊団
- **効果テキスト**:
  > 【ドン!!×2】【アタック時】②(コストエリアのドン!!を指定の数レストにできる),自分の手札1枚を捨てることができる：自分のライフが1枚以下の場合、デッキの上から1枚までを、ライフの上に加える。
- **能力分解（テキスト解釈）**:
  - 能力0 【ドン!!×2】【アタック時】: コスト=ドン!!2レスト＋手札1枚捨て / 効果=自ライフ≤1の場合、デッキトップ1枚**まで**をライフ上に加える / 対象=自デッキトップ / 数値=1枚まで(is_up_to) / 条件=ドン!!2付与 かつ 自ライフ≤1
- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |
  |---|----------|------|----------|
  | 1 | ドン2付与・手札1枚以上・自ライフ1 | アタック時発動 | デッキトップ1枚がライフ上へ(ライフ+1) |
  | 2 | 自ライフ3 | 発動 | 条件未達で効果なし(コストは支払い得る) |
  | 3 | 1枚までなので加えない選択 | 発動 | 0枚も可 |

- **パース照合**: condition=AND[HAS_DON GE 2, LIFE_COUNT LE 1]、cost=REST_DON+DISCARD(1)、effect=HEAL(value=1)。条件・コストは整合。ただし**effectがHEAL固定value=1で「1枚まで」(is_up_to)が落ちている**(0枚選択＝加えない選択ができない実装の疑い、HEALは常に1枚加える)。意味的影響は小さいが「までを」取りこぼし。

## OP03-099 シャーロット・カタクリ
- **基本**: 色=黄 / ライフ=5 / パワー=5000 / 属性=打 / 特徴=ビッグ・マム海賊団
- **効果テキスト**:
  > 【ドン!!×1】【アタック時】自分か相手のライフの上から1枚までを見て、ライフの上か下に置く。その後、このリーダーは、このバトル中、パワー+1000。
- **能力分解（テキスト解釈）**:
  - 能力0 【ドン!!×1】【アタック時】: コスト=なし / 効果=自分か相手のライフ上1枚までを見て上か下に置く → その後このリーダーにパワー+1000(このバトル) / 数値=1枚まで(is_up_to)＋1000 / 期間=THIS_BATTLE / 条件=ドン!!1付与
- **テストケース**:
  | # | 前提盤面 | 操作 | 期待結果 |
  |---|----------|------|----------|
  | 1 | ドン1付与 | アタック時、自ライフ上1枚見て下へ | 並べ替え後パワー+1000 |
  | 2 | 相手ライフを選択 | 相手ライフ上1枚見て上/下 | 相手ライフ操作後パワー+1000 |
  | 3 | 「1枚まで」で見ない選択 | — | スクライ無し、それでもパワー+1000 |
  - 境界/異常系: ライフ0枚側を選ぶと見る対象なし / 「までを」=見ない選択可
- **パース照合**: condition=HAS_DON GE 1。effect=Choice[自(LOOK_LIFE SELF→上/下)/相手(LOOK_LIFE OPP→上/下)/(空=見ない)] → BUFF(self,+1000,THIS_BATTLE)。「自分か相手」「上か下」「までを(見ない選択肢)」「その後+1000」全て表現されており整合。
- **実観測照合**: `ON_ATTACK: NO_CHANGE`。本来はINTERACTIVE(どちらのライフを見るかの選択)またはBUFF適用が期待される。ドン!!1条件が汎用盤面で未達のため発火せずNO_CHANGEと推定。条件さえ満たせば+1000は最低限EXECUTEDになるべきで、NO_CHANGEは条件未達由来とみる。
- **判定**: ⚠️ 要確認 — パースは整合。NO_CHANGEはドン!!1付与未達と推定だが、条件成立時に+1000とライフ操作が起きるか個別テスト要。
