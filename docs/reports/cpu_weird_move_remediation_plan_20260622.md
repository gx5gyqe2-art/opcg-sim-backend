# CPU「変な手」撲滅 計画メモ（測定駆動・近似パッチ撤去）（2026-06-22）

> 設計メモ（点）。実プレイで観測される CPU の「変な手」（防御の歪み・リソース浪費・自殺攻撃）を、
> 近似パッチの積み増しではなく **測定駆動で評価関数の根から是正し、補償パッチを安全に撤去する** 計画。
> 進捗の正本は WBS（`gx5gyqe2-art/WBS` の `projects/opcg-sim-backend.md`）。実装が固まった内容は
> `docs/SPEC.md §2.5` に吸収する（本メモは時点スナップショットとして改変しない）。

## 1. 問題設定

実プレイで残る「変な手」は次の3カテゴリに集約される（ユーザ観測・2026-06）:

- **防御の歪み**: 届かない部分カウンターを切る／温存すべき札を吐く／不要ブロック。
- **リソースの無駄遣い**: 無意味なドン付与・効果の空打ち・過剰展開。
- **自殺攻撃**: KO もトレードも割に合わない攻撃で攻撃体をレストに晒す。

この3つは別症状に見えるが、原因は一点に収束する＝**静的評価が「勝ちに変換されない要素」を ＋評価している**
（value-realization gap）。`docs/reports/cpu_precision_batch_20260616.md` の `normal<easy` 矛盾はその傍証。

## 2. すでに入っている対策（前提・再掲しない）

教科書的な対策は実装済みで、これらは**撤去対象ではない**:

- 静止探索 `_settle_eval`（相手 MAIN まで整流してから採点・`cpu_ai.py`）、防御の戦闘解決後採点
  `_score_defense_move`/`_settle_combat`（`cpu_mcts.py`）。＝**正しさ系（A）。恒久維持**。
- do-nothing 基準線（TURN_END 比較・`_ACT_MARGIN`）、無駄手の枝刈り `_prune_futile_attacks`/
  `_prune_don_moves`、ドン返し減点 `_W_DON_RETURN`、楽観是正 `W_SETTLE_PRESSURE`、telegraph lethal。
  ＝**挙動補償ハック（C）。最終的な撤去対象**（§7）。
- ビーム幅・killer・`TURN_ACTION_CAP`/`REPEAT_CAP` ＝**探索効率/安全弁（B）。維持**。

「対策済みなのに残る」原因は §1 の評価ズレに加え、(i) ホライズン切れで「過剰コミットのツケ」が
`W_SETTLE_PRESSURE`（受ける打点のみ）に拾われない、(ii) hard が相手手札を覗く完全情報（`see_opp_hand=True`）
ゆえ人間に不可解な“正解”を打つ、の2点が混じる。

## 3. 方針

**「物差しを作る → 切り分ける → 根を直す → 測りながら 1 個ずつ剥がす」。** 先に C を剥がすと、評価が
依然として悪手を ＋評価するため必ず退行する＝順序厳守。Phase 2 の野心度は **2a＋（教師あり
リキャリブレーション ＋ 1 回再生成）で確定**（線形モデルでは expert iteration の上積みが小さく、
インフラ費に見合わないため。フル 2b は昇格条件 §6 を満たしたときのみ）。

## 4. Phase 0 — 物差し（測定基盤）★最初の一手

「変な手」を主観から再現可能な数値へ。以降すべての合否判定器になり、Phase 2 の学習データ源も兼ねる。

- `tests/cpu_weird_move_audit.py`（新規）: 自己対戦 N 局で各決定の **null-move regret**
  ＝ `eval_settled(選択手) − eval_settled(TURN_END)`（両者 `_settle_eval` 経由＝戦闘解決後採点）を算出し、
  `差 ≤ 0 なのに行動を選んだ手`／自殺攻撃／無駄ドン／届かないカウンターをカテゴリ別に集計。
  さらに当該手以降を最後までプレイして **実際に負け筋へ繋がったか（結果ラベル）** を付す。
- `decide` トレース（`_fill_decision_trace`・TEST_SPEC §3.2）に null-move 項を追加。
- 既存資産流用: `tests/collect_value_data.py`（自己対戦収集）・`tests/cpu_arena.py`（凍結ベースライン Elo）・
  `tests/test_cpu_puzzles.py`（決定論パズル）。
- **合格ゲート**: 3カテゴリの「変な手カウンタ／100局」をベースライン値として確定＝以降の基準線。
  代表局面を `test_cpu_puzzles.py` に凍結（逸話→回帰テスト）。

## 5. Phase 1 — 切り分け＆汚染源除去（評価 / ホライズン / カンニング）

`cpu_arena.py` の ablation A/B（席交互・hard・既存 [A2] 手法を踏襲）で 3 症状の主因を特定し、
**学習前に汚染源を除去**する（Phase 2 のデータを汚さないため）。

1. **評価仮説**: `W_ATTACKER`/`W_FIELD_POWER`/`W_DON_ACTIVE`（`cpu_ai.py`）を半減 → 無駄展開・自殺攻撃が減るか。
2. **カンニング仮説**: hard の `see_opp_hand` を expert 同様に **手札決定化** して比較 → 防御の歪みが減るか。
3. **ホライズン仮説**: `W_SETTLE_PRESSURE` を「受ける打点」だけでなく **過剰コミット盤面の損
   （レスト攻撃体／付与キャラの被 KO）** も含むよう拡張 → 自殺攻撃・ドン浪費が減るか。

- **合格ゲート**: 各症状の主因を数値で確定。勝率＋Phase0 カウンタの両方で判定。汚染源（カンニング・
  探索バグ）を是正してから Phase 2 へ。

## 6. Phase 2 — 評価のキャリブレーション（2a＋ 確定）

手調整の沼から脱出し、評価を実勝率へ寄せる。既存スキャフォールド（`cpu_features.py`/
`tests/collect_value_data.py`/`tests/train_value.py`/`value_model.json`/`OPCG_VALUE_BLEND`）を実行・拡張する
＝WBS「評価関数の学習化」の未着手項（自己対戦 Elo で α・反復数調整 → 安全導入）の本実行。

- **サブ0（前提）特徴カバレッジ監査**: `cpu_features` が手作り評価の概念（非線形ライフ／カウンター価値／
  脅威キーワード／ドン／盤面パワー／デッキ危険域）を網羅するか確認。欠ければ**特徴追加が先**
  （無い概念は学習で直せない）。
- **データ生成 v1**: Phase 1 で汚染源を除去した AI で自己対戦 →（特徴, 最終勝敗）を JSONL 収集。
  ラベルは Monte Carlo（win=1/loss=0）。特徴は運用情報状態に合わせ算出（既定＝公開情報ベース・
  hard の相手手札透視を学習評価に混ぜない）。
- **学習 v1**: pure-Python ロジスティック回帰（stdlib-only）で `value_model.json` 出力。検証＝ホールドアウト
  log-loss／キャリブレーション曲線／決定論一致。
- **ブレンド導入**: `OPCG_VALUE_BLEND` を段階的に上げ、`cpu_arena` で勝率＋Phase0 カウンタを測定し最良 α を選定。
- **再生成（2a＋ の肝）**: 最良 α の改善評価で **もう一度データ生成→再学習**（分布シフト分を 1 回だけ回収）。
  良ければ採用・悪ければ v1 へ差し戻し。
- **合格ゲート**: 固定基準 AI に対し **勝率 > 55%** かつ Phase0 カウンタが基準線比で有意減、現行 hard/expert で
  `normal<easy` 的矛盾なし。
- **2b 昇格条件（満たすときのみ）**: ① 2a＋ で勝率が目標未達 ② かつ Phase0 カウンタは既に低い（残差が
  “弱さ”であって“奇行”でない） ③ NN 等の容量増強を投資する覚悟がある——3つ揃ったときだけ expert iteration へ。

## 7. Phase 3 — 補償パッチ（C）の撤去

評価が賢くなった分、C を **1 個ずつ測って剥がす**。各パッチごとに: ①「知識」が評価に吸収済みか確認 →
② パッチ OFF の A/B（`cpu_arena`）→ ③ **合格条件＝勝率非劣化（信頼区間内）＋ 該当カテゴリの Phase0
カウンタが増えない** を満たせば撤去・満たさねば差し戻し。

- **撤去順（依存の浅い順）**: `_W_DON_RETURN` → `_prune_don_moves` → `_prune_futile_attacks` →
  脅威キーワード手盛り → `_ACT_MARGIN`（最後・良評価なら margin→0 の安全弁へ格下げ）。
- **残すもの**: A（`_settle_eval`/戦闘解決後採点）と B（beam/killer/各 CAP）は恒久維持。

## 8. Phase 4 — 品質ゲート & SPEC 吸収

- 全テスト pass（`OPCG_LOG_SILENT=1 python -m pytest tests/ -q -s -p no:cacheprovider`）。
- 構造監査 0（`python tests/full_card_audit.py`）。
- 挙動を意図的に変えた箇所は `full_card_audit.py --regen` でベースライン再生成＋差分レビュー。
  検証済みデッキの挙動変化は `tests/test_verified_decks.py` に追記。
- 確定内容を `docs/SPEC.md §2.5` に吸収・`TEST_SPEC.md`（§3.1/§3.2）を更新。各 Phase の計測は `docs/reports/`
  に日付付きで追記（改変禁止）。

## 9. 依存関係・マイルストーン

```
Phase0 物差し ─→ Phase1 切り分け＆汚染源除去 ─→ Phase2(2a＋) 評価学習 ─→ Phase3 パッチ撤去 ─→ Phase4 ゲート
            └→ 以降すべての合否判定に常用
```

- Phase 2 は Phase 0（ラベル付きデータ＋判定器）と Phase 1（クリーンなデータ）に依存＝着手順は固定。
- **M1** Phase0 完了＝変な手の数値ベースライン確定 ／ **M2** Phase1 完了＝主因特定＆汚染除去 ／
  **M3** Phase2 完了＝勝率>55% & 矛盾解消 ／ **M4** Phase3 完了＝C パッチ最小化。

## 10. 主なリスク

- 学習データ汚染 → Phase 1 で汚染源を先に潰す（順序厳守）。
- アリーナの分散（±35 Elo 程度のノイズ帯が既知）→ 十分な試行数と信頼区間で判断。
- value-model 評価のレイテンシ → `OPCG_VALUE_BLEND` の段階導入＋PyPy ワーカー経路で吸収。
