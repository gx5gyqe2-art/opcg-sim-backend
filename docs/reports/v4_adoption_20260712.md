- 学習損失: val MSE 0.38→0.011 まで単調下降・発散なし。補助ヘッドは ±1.0〜1.5T で推移。

## 3. 既知の限界（次期 run への引き継ぎ・ユーザ承認済みのトレードオフ）

1. **攻めの付与ニュアンスの退行**: 「ドン付与→攻撃」が正着の局面（g1@12 等）で付与を選ばない
   （v3 の 0.80 → v4 の 0.00）。防御獲得（付与抑制・カウンター）の裏返しの過剰一般化で、
   10k までの追加学習でも自然回復しなかった＝**構造的**（付与が報われる系列がデータで薄い）。
   → 次期 run で **マーク局面シード（旧計画 §4-1(e)・未実施のまま）** ＋付与系マークの蓄積で対応。
2. **ピーク一過性**: 継続学習の忘却は混合ラベルでも残る。次期はピーク検出の自動化
   （mark_gate/対v3 の同時後退でアラート）と、忘却抑制の損失設計（EWC/蒸留正則化等）が論点。
3. 時計誤差の対面別偏り（ワースト対面 ±3〜4T）: 自デッキ残の集約特徴（旧計画 §5.5-2）が
   次期 encoder v4 候補。

## 4. 運用知見（次の run のために）

- **セッションのコンテナはアイドル回収される**: 生成ワーカー/learner をバックグラウンド常駐させる
  セッションは、エージェント自身が10分間隔の生存確認ループを張ること（keep-alive）。learner は
  司令塔セッション（会話で常時アクティブ）に置くのが最も安定した。
- **小バッチ（--games 32）＋consume/学習の分離（wave 蓄積）**: セッション断のロスを最大32局に抑えつつ
  games:updates 比を保つ。pending は manifest 永続化＝learner の停止・移譲・再開が安全。
- 5ワーカー並列は帯域十分だがセッション断が頻発（実効3本で 10k/約半日）。本数より keep-alive が効く。
