# 凍結設計 v4b（最終）: レビュー第7巡の2つの致命傷修正＝実装移行条件の合意

日付: 2026-07-01 / `cpu_rl_frozen_design_v4_20260701.md`＋`v4a` への最終修正。レビュー第7巡
（「この2点を修正できれば死角なし・実装移行を承認」）を反映。**本書の合意をもって設計凍結・実装フェーズ移行**。

> スナップショット（改変しない）。

## 1. 修正1: 動的「相手隠匿プール集約統計」を破棄 → 「リーダー所属色マルチホット」を復活
v4a の「プール挙動要約」（相手の手札+山札の集約fingerprint統計）を**破棄**する。理由（採用した反証）:
- **カウンティングへのドリフト**: ゲーム進行で母集団が縮み、統計が「残り山札のカウンター枚数」という
  確定的未来へ変貌する。ネットは駆け引きでなくカンニングペーパーに過剰適合し、しかも特徴の
  「精度」が非定常（序盤=事前分布・終盤=確定情報）＝1セット学習と最悪の相性。
- **決定打（本レビューで確定した判定基準）**: **リーダーは公開カード＝その色は純粋な公開情報**。
  一方プール統計は（初期デッキリスト集計に静的化しても）「相手デッキリストを特徴として知っている」
  前提を要する。**特徴に入れてよいのは公開情報のみ／隠匿情報の仮定は PIMC の探索内に閉じ込める**、
  という一線を引く。
- 質的解像度（黒=コスト参照除去/青=バウンス/黄=ライフ操作で対処が180度違う）も色が担う。

**採用形**: グローバル状態ベクトルに**自分と相手のリーダー所属色マルチホット（2×6次元）**を復活。
静的（ゲーム中不変）・低次元・公開情報。**カードごとの色特徴は廃止のまま**（アブレーションで
実証した shortcut は per-card 色の話であり、グローバルなリーダー色メタ情報とは層が異なる）。
リーダー自体の振る舞いは既にその fingerprint（公開カード）で符号化済み。

## 2. 修正2: PIMC 決定化の「ライフ据え置き（透視）」を Gen0 前の必須修正（Blocker）に昇格
現行 `_determinize_opponent` は相手の手札+山札のみ再サンプルし、**ライフは実物のまま**＝探索が
ライフ内トリガーを透視する。自己対戦学習に持ち込むと、透視前提の方策が生成した軌跡から
「トリガーを踏まない世界」の楽観 value を学習し、探索外で value が呼ばれた時に不完全情報下の
リスク評価が崩壊する（**探索時の情報優位＝価値関数の汚染**）。「将来課題」扱いを撤回し **Blocker** とする。

**採用形（学習系の決定化のみ変更・L1 は不変）**:
- 相手: **手札+山札+裏向きライフ** の合同プールから再サンプル（手札枚数・ライフ枚数は保存）。
- 自分: **山札順+裏向き自ライフ** も再サンプル（自分も自ライフの中身は知らないのがルール。
  自手札・場は実物のまま＝返す手の合法性は不変）。
- **表向き（公開済み）ライフは再サンプルしない**（公開情報）。
- 適用範囲: 学習型 CPU の determinize（本番 `learned/adapter.py`・訓練 `tests/opcg_game.py`）と
  自己対戦ループ。**L1 の PIMC は触らない**（ゲート基準・製品挙動の安定のため。L1 の据え置きは
  静的評価+α-β の既知の妥協として存置）。
- 検証: 修正後に pre-flight（脱もつれ×MCTS vs L1）を再実行し、数字が崩れないことを確認してから
  Gen0 へ（公平化で見かけの強さが下がる可能性はあるが、それが「正しい強さ」）。

## 3. 凍結（最終）
v4/v4a の他の要素（held-out実デッキゲート＋SPRT・エンジン駆動型パラメトリック生成・
二部サンプリング α・機序ペアインデックス・Early Fusion・密補助損失＋巨大スイング項・
checkpoint/rollback・Gen2 fallback）は不変。レビュー第7巡はこの2点の修正を実装移行条件として
承認済み。**以降は実装フェーズ**（held-out freeze → インデックス/クラスタ → 生成器 →
決定化修正 → 新ゲート実測 → Gen0）。
