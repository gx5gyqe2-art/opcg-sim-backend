# 学習型OPCG CPU「1セット汎化」凍結設計 v4（目的再定義＋生成方式改訂・レビュー第5巡反映）

日付: 2026-07-01 / **v3 を supersede**。契機はユーザの2指摘（「色って本当に測る必要があるの？」
「実デッキ学習はそのデッキに詳しくなるだけでは？」）と外部敵対レビュー第5巡。

> スナップショット（改変しない）。

## 0. v3からの変更点（要旨）
1. **ゲート再定義**: 主ゲート＝黄ゼロショット → **held-out実デッキリスト vs L1**。
   カードプール2652種は全既知＝本番で未見の色/カードは構造的に発生しない。未知なのは
   「既知カードからどう50枚組まれるか」だけ。色ゼロショットは目的より難しい人工極限だった。
2. **訓練デッキ生成の改訂**: 実リスト種ミューテーション → **エンジン駆動型パラメトリック構造生成**
   （実リスト不参照・下記§2）。実デッキは**テスト専用**（訓練に使うと近傍暗記の確認＝リークになる）。
3. **①b色除去は「必須の制約」に復帰**（v4案の「安い保険」格下げをレビューが棄却）: 全色訓練でも
   プール自体の色偏り（黄=トリガー多い等）があり、怠惰なネットは fingerprint を読むより
   「色→アーキタイプ強度」の1次元shortcutを確実に見つける。実測でも色除去は in-dist さえ改善した。
4. **色holdoutは常設の軽量ベンチマークに**（記憶化カナリア）: 「特定シナジーパッケージを丸ごと抜いた
   転移精度」を checkpoint 評価に常設し、将来のプール拡張時の表現ロバスト性の基準を残す。

## 1. ゲート③（受け入れ基準）
- **held-out実デッキ集合をfreeze**（ユーザの黒ひげ実構築を筆頭に数個）。訓練は全色・全カードOK、
  held-outの「リスト」だけ除外。
- **合格ライン: vs L1 勝率 > 0.60**（ルールベース限界の突破）。**SPRT（逐次確率比検定）で早期終了**
  （明確に0.70超え/0.40未満なら100戦を待たず打ち切り）＝評価コスト律速を回避。
- 最終ゲートの相手は**製品強度のL1**（greedy-L1でなく α-β+PIMC。研究中の代理はgreedy-L1可）。

## 2. 訓練デッキ生成②: エンジン駆動型パラメトリック構造生成
**役割**: 汎化の主エンジンは振る舞いfingerprint表現。生成多様性は「デッキ固有の暗記を許さない圧力」。
毎ゲーム新規サンプル＝デッキ識別の学習を構造的に不可能にする。

### 生成手順
1. **コアパッケージのシード（エンジン駆動・実リスト不参照）**: リーダーを選び、
   「リーダーとシナジーする中核カード数枚」を**カードDB構造から**導出して固定
   （リーダーのTrait保持者＋そのTraitを効果で参照するカード等）。OPCGは「リーダー中心の
   非線形コンボのゲーム」であり、純パラメトリックだけでは「歪んだが強力な構築」が分布から欠落する
   （レビュー第5巡の主要指摘＝ドラフト環境と構築環境のドメインギャップ）。
2. **残り30〜40枠をパラメトリック充填**: 構造パラメータ（コストカーブ形状/カウンター密度/
   トリガー密度/ブロッカー数/Trait集中度/効果カテゴリ配合/色数）を空間からサンプルし、
   **事前分類した役割クラスタ**（fingerprintの事前クラスタリング・オフライン1回）から一様サンプル。
   毎ゲームのナップサック的厳密解は解かない（CPU負荷・グッドスタッフ化の回避）。
3. **Trait集中度の数理（両側インデックス）**: パーサは `TargetQuery.traits` を構造化済み
   （効果でTraitを参照するカード433枚）。オフラインで **trait→保持者** と **trait→参照者
   （サーチ/バフ）** の2インデックスを構築。集中度λ＝「保持者∪参照者から引く割合」として
   O(1)サンプリングで制御（保持者だけ集めてもシナジーは機能しない、の解）。
4. **プレイアビリティ監査**: L1同士で規定内決着・Q値に勾配・Action-Advantage分散。
5. **極端ノイズの常設**: 数%は「極端だが合法」構成（バウンダリのハードコード下限）。

### リーク防止（実デッキの扱い）
- 実デッキは (a) ゲート専用 (b) **Covering Radius監査＝「生成分布が実デッキを覆うかの一方向確認」のみ**。
  監査結果で生成分布を**絞る調整はしない**（絞ると凸包内に閉じ込められアウトライア耐性が死ぬ＝
  マクロなメタデータリーク）。生成バウンダリは実デッキ統計より**意図的に±20%以上広く**設定。

## 3. 表現・学習（v3から継続＋補強）
- 効果fingerprint（**raw色除去=必須**）＋エンジン注入の決定的特徴（無効化0/1・有効対象数・
  発動条件距離・メタ常在フラグ）のEarly Fusion。ID埋め込みなし。Trait関係は二値マスク。
- MCTS必須・ポインタ方策・L1模倣warm start（value/policy）＋KL制約・self-play outcome。
- 密補助損失: パワー差/ライフ差予測 ＋ **「1ターン内の巨大な盤面スイング」予測項を追加**
  （構築デッキの非線形な理不尽への警戒を強制学習＝ドメインギャップ緩和・レビュー第5巡）。
- フェイルセーフ: checkpoint+rollback / マイクロセット感度分析 / Gen2 fallback（不変）。

## 4. 実行順
1. held-out実デッキ集合のfreeze（黒ひげ実構築の再構成＋数個）
2. 役割クラスタ＋trait両側インデックス（オフライン）→ エンジン駆動型パラメトリック生成器
3. プレイアビリティ/Covering Radius監査（一方向）
4. 新ゲート（held-out実デッキ vs L1・SPRT）で「生成訓練＋脱もつれ表現」の効果を実測
5. GREEN→本走（outcome self-play・世代ごと checkpoint評価にカナリア常設）

## 5. 最も失敗しやすい1点（レビュー指摘・監視項目）
**生成環境（ドラフト的・線形な駆け引き）と実デッキ環境（構築的・非線形スイング）のドメインギャップ**。
緩和: コアパッケージのシード（§2-1）＋スイング予測補助損失（§3）。ゲートで実測監視。
