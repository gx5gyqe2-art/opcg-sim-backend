# P3 v2 本走：エフェメラルな Claude Code セッションでの実行記録（2026-07-05）

`p3_run.py` の設計意図（「揮発性環境で忠実な本走を完遂」）が、**常設CPU VMではなく本セッションと同種の
エフェメラルなクラウド実行環境（Claude Code remote session・コンテナ）でも実際に成立する**ことを、
v1本走の遺構調査＋v2本走の実行を通じて実証した記録。既存ドキュメント（`p3_run.py` 冒頭コメント・
`cpu_rl_pilot_p3_harness_20260629.md`）は「常設CPU VM前提」「揮発性devでは不可」と書かれているが、
これは**目標状態の推奨**であって**必須要件ではない**——という訂正・補足を兼ねる。

## 1. 環境

- Claude Code remote session（本レポート作成と同一種）。`nproc`=4・`free -h`=15Gi。永続VMの明示的な
  provisioning は無い＝セッション/コンテナはいつ再起動されても不思議ではない前提で運用した。
- 実際、本走の途中で**コンテナ再起動が1回発生**した（§4）。

## 2. v1本走の遺構調査（本走に先立って発見）

`claude/p3-checkpoints` ブランチに、**この種の環境で過去に実行された v1 本走**の最終状態が残っていた:

```json
{"gen": 2, "cum_games": 0, "shards": 337, "status": "AWAITING_GATE"}
```

`gen2_value.npz`/`gen2_policy.npz` は SHA256 が現行出荷 `opcg_sim/data/learned/gen2_*.npz` と**完全一致**
＝**今の製品CPUは、まさにこの環境種別で実行された本走の成果物**であることが裏付けられた。337シャード
（`--shard-games 60` なら 20,220局＝2世代分）は、周辺のdevコミット日時（2026-06-18〜2026-06-29・
約11日）から、**継続実行ではなく多数回の分割実行**で積み上げられたと推定される（`push_ckpt` が
shardごとに `commit --amend`+`force push` するため、個々のshardのタイムスタンプ自体は残らない）。

v2本走のため、このチェックポイントは `claude/p3-checkpoints-v1-archive`（force-push無しの単純複製）
へ退避した上で、通常コミット（fast-forward・force不要）でクリーンな `{gen:0,cum_games:0,shards:0,
status:INIT}` へリセットした。

## 3. v2本走の実行

```bash
OPCG_LOG_SILENT=1 PYTHONPATH=tests python tests/scripts/p3_run.py \
  --enc-version 2 --rotate-leaders --shard-games 60 --sims 40 --workers 4 --max-shards <N>
```

- **温スタート**: v2 Gen0 は出荷 v1 Gen2 から拡張（増えたスカラー特徴のゼロ行挿入）＝拡張直後の出力は
  出荷 v1 と恒等（実局面で `|Δ|` 1e-9〜4e-16 で確認）。乱数初期化ではなく出荷の実力から開始。
- **チャンク分割**（`--max-shards 12〜20`）を**バックグラウンドBashで繰り返し起動**する運用にした:
  1回の呼び出しは有限個のshardで自然終了 → 完了通知を受けて進捗確認 → 次のチャンクを起動、を繰り返す。
  各shardは完了ごとにgit checkpointブランチへコミット・pushされる設計（`push_ckpt`）なので、
  **チャンクの区切り自体はどこで区切っても進捗を失わない**（連続1invocationでやる必要が無い）。
- 進捗の目安（実測）: 1シャード（60局・sims=40・4並列・クローン版MCTS）**約245秒**。

## 4. 実行中の変更投入（別セッションからのHANDOFF 3件）

本走の途中で、**別セッションが同リポジトリで作業した改修**を取り込む必要が生じた:

1. **make/unmake版MCTS**（`OPCG_MU_MCTS=1`・clone廃止で自己対戦高速化）
2. **学習MCTSスタックの重複解消**（`tests/harness/*` を本番 `opcg_sim/src/learned/*` への委譲shimに統一）
3. **探索オプション既定の config.py 集約**（2の追記）

### 4.1 mu-mcts の取り込みと切替

- 対象PRが既にマージ済みだったため、規約どおり指定ブランチを最新mainから作り直し（`git checkout -B
  <branch> origin/main`）→ 該当2コミットを cherry-pick。
- 検証: パリティ（clone版とdeterministic局面で完全一致 6/6）→ 全テスト1161 pass → slow(journal/
  make-unmake) 4 pass → 構造監査0。速度計測は**本走と同時実行だと競合で数値が歪む**（実測2.38×、
  期待2.7〜3.6×）ため、額面どおり受け取らず「原因の分かる逸脱」として処理し、投入自体は進めた。
- **切替タイミングはユーザ判断で「次のチャンクから」**（世代境界を待たない）。理由: shard単位で
  checkpointされる設計上、チャンクの入れ替わり地点で切り替えても失うものが無く、mu-mcts自体も
  「clone版と同一意味論・別RNG実現」（handoff指定のper-simリセットで保証）なので、世代内で
  混在させても実害が薄いため。
- **実測の効果**（本走との競合が無い状態の2チャンク・計40shard平均）: **約126〜134秒/shard**。
  クローン版平均245秒からの実速度向上は**約1.9〜2.2倍**（隔離ベンチの2.7〜3.6倍より控えめ）。
  差分は主に固定オーバーヘッド（shardごとのgit fetch/reset/push・プロセス起動）で、MCTS自体が
  速くなっても短縮されない部分と考えられる。

### 4.2 重複解消＋config集約の投入

両handoffとも明示的に **「本走に影響するので投入は世代境界（AWAITING_GATE）で」** と指示していた。
これを次のように解釈・実行した:

- **開発・検証は隔離した別git worktree/別ブランチで行い、ライブの学習チェックアウト
  （バックグラウンドチャンクが読むファイル）には一切触れない**。全ゲート（全テスト1161・
  targeted 34件＝GATE B単調性含む・slow 4件・構造監査・ruff）を隔離ワークツリー側で通した。
- **投入（ライブブランチへの合流）はユーザの明示判断で「次のチャンク境界」**に前倒しした
  （フルゲート境界=AWAITING_GATEを待たなかった）。根拠: 検証が徹底的（挙動不変を数値レベルで
  確認済み）で、投入自体は fast-forward（分岐なし）、投入後も軽量smoke（30 pass）で再確認。
  「世代境界で」という指示は安全側に倒した推奨であり、検証済みの無害な変更を早める判断は
  ユーザ自身が行った。

## 5. コンテナ再起動からの回復（実地検証）

チャンク実行中に**コンテナが再起動**し、実行中だった `p3_run.py` プロセスと、隔離ワークツリーでの
作業が一時的にリスクにさらされた。実際に起きたこと:

- **学習の進捗は無傷**: 再起動直前の完了shardのcheckpoint（`cum_games=7020, shards=117`）が
  そのまま保存されていた。失われたのは着手直後だった次のshard1つぶんのみ（実質数分）。
  → `p3_run.py` の設計目標「回収で失うのは直近1シャードのみ」が**絵に描いた餅ではなく実際に機能する**
  ことを確認できた。
- **ローカルディスク（/tmp含む）とPython環境（numpy/pytest）は再起動を跨いで保持**されていた
  （このセッションでは）。ただし、**未pushのgitブランチはローカルにしか無い間はリスクに晒される**
  ため、このインシデント後は**隔離ワークツリーの一時ブランチも即座にoriginへpush**する運用に変更した
  （fast-forwardで合流させる前提なので、pushしても本流を汚さない）。

## 6. 得られた実務上の知見（次回この種の環境で回すときのために）

1. **常設VMは必須ではない**。「小shard＋shardごとgit checkpoint」設計が volatile 前提そのものを
   吸収するため、エフェメラルなクラウドセッションでの分割実行で本走は完遂できる（v1・v2とも実証）。
2. **1つのcheckpointブランチに対しては同時に1invocationのみ**。並列に回すと共有worktree
   （`/tmp/p3ckpt-wt`）とforce-pushが競合する。
3. **BLASスレッドは`--workers`実行前に1へ固定**（`OMP_NUM_THREADS`等）。既にp3_run.py側で対応済みだが、
   環境変数の設定漏れがあると4プロセス×フル並列でスラッシングする。
4. **同時に重いテスト/ベンチを走らせると速度計測が歪む**。正しさの検証（テスト・パリティ）と
   速度計測（実shardタイム）は別のタイミングで行う。
5. **本走中の危険な変更（挙動に影響しうる改修）は隔離ワークツリー＋別ブランチで検証してから
   fast-forward合流**する。ライブチェックアウトのファイルを直接いじらないことで、バックグラウンドの
   チャンクを止めずに並行開発できる。
6. **一時ブランチも早めにoriginへpush**しておく（ローカルのみの資産はコンテナ再起動のリスクに晒される。
   fast-forward前提なら気軽にpushして問題ない）。
7. **世代境界（AWAITING_GATE）待ちは安全側のデフォルトであって絶対ではない**。検証が十分（挙動不変を
   数値で確認済み）なら、ユーザ判断でチャンク境界に前倒しできる。
