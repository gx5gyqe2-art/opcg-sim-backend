# 品質ポストモーテム — カテゴリH（先頭条件が「。その後、」をまたいで漏れる）の見逃し

> 種別: 報告（点）。2026-06-14 時点のスナップショット。以後の改変はしない（履歴として残す）。
> 関連: `docs/TEST_SPEC.md` §8.2 台帳 / §8.3 バグ類型カタログ / §8.6 検証計画。

## 1. 事象

EB/PRB/P 弾×色検証の過程で、**「【…】〈条件〉の場合、A。その後、B」という構造で、先頭の条件が
`branch` として A だけを包み、後続 B（登場/KO/レスト/付与/ドンアクティブ等の実効果）が条件の外に
出て無条件実行される**系統的バグを発見した（仮称: カテゴリH）。

OPCG ルールでは先頭に置かれた条件は能力全体（その後 B を含む）を支配するため、これは誤り。

代表例:
- **EB02-032 アイスバーグ**: 「ドン!!3枚以上ある場合、…見て手札に加える。その後、手札からガレーラを**登場させる**」→ ドン<3でも登場
- **EB03-017 ジュエリー・ボニー**: 「リーダーが超新星の場合、ドンをアクティブ。その後、相手キャラを**レスト不可に**」→ 超新星でなくても適用
- **OP04-033 マッハバイス / OP04-026**: 百獣でなくても「その後」のドンアクティブが発火
- **ST11-004 新時代**: 「ウタの場合…加える。その後、…ドン1枚アクティブ」（ウタ専用＝実害は小）

## 2. 規模

genuine（TEMP/デッキ整理だけの無害な後続を除く）で **119能力・全弾**。
うち **74能力（62%）が OP05〜OP16 ＝「弾×色検証済み・✓ラチェット済み」とした弾**に存在していた。
「検証済み」ラベルの過半が素通りしていたことになる。

| 区分 | 件数 |
|---|---|
| OP05〜OP16（検証済み弾） | 74 |
| OP01〜OP04 | 15 |
| EB/PRB/P | 22 |
| ST | 4 |
| 合計 | 119（＋非index-0 の OP11-066 ほか） |

## 3. なぜ見逃したか（多層の死角が同じ場所で重なった）

### 3.1 近接原因：カテゴリA修正の設計境界
カテゴリA は「〈条件〉場合、A**して**B」＝**1文内の連用接続**を対象に、条件を ability レベルへ lift して
全体をゲートした（OP15-104/OP09-005/OP09-024 等は正しく修正済みと確認）。だが設計上
**「。」「その後、」等の手順境界は従来どおり分割**した。この結果、**先頭条件が文1にあり「。その後、」で
文2のBが続く形（H）は構造的に未処理のまま残った**。A と H は字面が酷似するため「条件分岐＋後続は
解決済み」という誤った包括感を生んだ。

### 3.2 ベースラインは“バグごと凍結”する（検出不能の本丸）
`full_card_baseline.json` は各能力を汎用盤面で1回走らせた指紋。Hのバグは**条件が偽のとき B が走る**
ことだが、ベースラインは条件成否に関わらず engine の出力を正本として固定する＝**漏れた B が
「正しい挙動」として焼き込まれた**。ベースラインは「変化（退行）」しか検出せず、**正しさ自体は
検証しない**。よって差分も出ない。

### 3.3 オラクルゲートは別の軸しか測っていない
HAS_OTHER／PER_TURN_LIMIT_GAP／UP_TO_GAP=0 は**パーサのフォールバック量**の指標。Hは綺麗に
パースされる（OTHERにならない）ので**ゲートは常に緑**。条件の*スコープ*は測定対象外。

### 3.4 classify 観測も素通り
classify は実行して state delta を出すだけ。漏れた B の盤面変化は**意図した変化と区別がつかない**。
「条件スコープ外の実行」を示す WARN は無い。

### 3.5 §8.3 カタログ「条件の退化」は“中身”の検査で“範囲”は対象外
既存カタログは条件の type/player/operator/value の正しさを見る。Hは**条件自体は正しく、被覆範囲
（どのアクションを覆うか）だけが狭い**。検査軸がそもそも無かった。

### 3.6 トリアージのラベル混同 ＋ ノイズ埋没
弾×色トリアージの危険パターン「条件分岐+後続」が **A（修正済み）と H（未修正）を同一ラベルで一括**。
フラグされた A型カードを健全と確認 →「このパターンは大丈夫」と一般化した。さらに「その後、残りを
デッキの下に置く」型の**無害な TEMP 整理が多数**で、実害ある少数（登場/KO/レスト）が埋没した。
今回 H を分離できたのは、**AST 構造不変条件**（先頭 branch かつ非TEMPの後続）という別の検出器を
新規に当てたから。

## 4. 一言でいう教訓
**「検証済み✓」のラチェットは“退行しない”保証であって“正しい”保証ではなかった。**
ベースラインは latent bug を凍結し、オラクルは別軸を測り、人手トリアージは類似パターンを一括処理
した — 三者の死角が同じ場所で重なった。

## 5. 横展開（同種の見逃しが他に無いかの調査）

カテゴリH と同じ性質（**ベースラインが凍結し、オラクル/classify が盲点**になる構造的バグ）を持つ
候補クラスを、構造不変条件スキャンで全弾横断調査した。

| スキャン（構造不変条件） | 結果 | 判定 |
|---|---|---|
| **Duration write-off**（「このターン中/まで」等の時限テキストがあるのに BUFF/付与/KEYWORD が `duration=INSTANT`） | 0件 | ✅ クリーン |
| **chooser 欠落**（「相手は自身の…捨てる/戻す/置く」で対象選択者が controller のまま＝相手の隠匿札を自分が選べる） | フラグ24件はすべて誤検知（実アクションは `chooser=OPPONENT` 設定済み。RETURN_DON 系は OP16-074 の責任者視点 resume でカバー。Choice 内ノードも設定済み） | ✅ クリーン |
| **「すべて」count 退化**（テキストに「すべて」があるのに対象 count≥1・select_mode≠ALL） | フラグ16件はすべて誤検知（「すべて」は正しく ALL／フラグは同カードの別アクション）。OP13-064 全体効果無効は既知残（§8.2） | ✅ クリーン（既知残を除く） |
| **H-variant（非 index-0）**（先頭が無条件 LOOK 等→分岐→その後 で漏れ） | 3件。OP04-011 ナミ・EB01-029 は「公開したカードを無条件でデッキ下」で**無害**。**OP11-066 シャーロット・オーブン**は「当たりならKO。その後、ドン1枚追加」でドン追加が条件外に漏れる＝**カテゴリH と同根の実害**（index-0 検出器の取りこぼし） | ⚠️ H に内包（要・検出器拡張） |

**結論**: カテゴリH 以外に**新規の系統的見逃しクラスは検出されなかった**（Duration・chooser・すべて は
健全）。ただしカテゴリH の真の範囲は index-0 限定の119能力より僅かに広く、**非 index-0 の漏れ
（OP11-066 等）も含む**。修正時の検出器は「先頭の無条件セットアップ（LOOK/宣言）に続く分岐」も
対象に含めること。

## 6. 再発防止策

1. **構造不変条件をゲート化（最優先）**: 「先頭（またはセットアップ直後）が `branch(if_false=None)` の
   能力に、非TEMPの後続兄弟アクションがあってはならない」を `effect_oracle` 系のラチェット（上限0）
   として追加。**カテゴリH修正後に0へ落とし、以後の再混入を機械的に防ぐ**。これがあれば初回検証で
   検出できた。本ポストモーテムの横展開で使った4スキャンも回帰ツール化を検討する。
2. **条件“偽”パスの被覆**: 検証/回帰で**条件を偽にして実行し「状態変化ゼロ」を確認**する観点を
   §8.4 チェックリストへ追加（現行ベースラインは真パスしか踏まない死角）。
3. **トリアージのラベル分割**: §8.3「条件分岐+後続」を **A（文内連用）／ H（文跨ぎ「。その後、」）** に
   分離し、別々に精査する。
4. **ベースラインの位置づけ明確化**: ベースラインは“安定性”の保証であり“正しさ”の保証ではない、と
   §8.5 に明記済みの方針を改めて運用に徹底する。正しさは意味的回帰（`test_verified_decks.py`）か
   構造不変条件で担保する。

## 7. 対応状況
- カテゴリH は §8.2 台帳・§8.3 カタログ・§8.6 進捗・WBS 残課題に記録済み。修正（先頭分岐で能力
  全体をゲート＋ベースライン再生成 ~119カード＋構造ゲート追加）は別途まとめて実施する。
- 横展開で確認した Duration/chooser/すべて は健全のため対応不要。
