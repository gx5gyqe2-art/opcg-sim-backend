# CPU 強化（強さ=Elo 優先）ロードマップ メモ（2026-06-22）

> 設計メモ（点）。CPU の主目的を **強さ（凍結ベースライン hard に対する Elo）** に置き直す。
> 旧「変な手撲滅」計画（体感品質が主目的・本メモ作成時に撤去）を、強さ優先へ転換した版。
> 進捗の正本は WBS（`gx5gyqe2-art/WBS` の `projects/opcg-sim-backend.md`）。実装が固まった内容は
> `docs/SPEC.md §2.5` に吸収する（本メモは時点スナップショットとして改変しない）。

## 1. 目的の再定義

- **主指標 = Elo**（`cpu_arena` の凍結ベースライン hard に対する勝率／Elo・席交互）。
- **副指標 = 変な手カウンタ**（null-move regret 監査）。強さに効く範囲でのみ気にする（格下げ）。

旧計画は「変な手（防御の歪み・リソース浪費・自殺攻撃）の撲滅」=体感品質が主目的で、Elo は従属ゲート
だった。本ロードマップは強さそのものを主目的に据える。

## 2. 最重要の認識 — 探索と評価は一緒に上げる

一般ゲームAIの定石として、**強さは「探索の質 × 評価の質」の積**で決まり、片方だけ伸ばすと壁に当たる。

- 評価には **value-realization gap**（勝ちに変換されない要素を ＋評価）があると実測済み。
- バイアスのある評価のまま**探索だけ深くすると、地平線でバイアスを増幅する**（horizon amplification）
  ＝深く読むほど"評価の勘違い"を確信を持って実行し、Elo が頭打ち/退行しうる。
- 逆に**探索効率（置換表）を上げずに評価だけ直しても深さが伸びない**。

旧計画 Phase0/1 の所見の射程に注意:
- `search_dispreferred = 0`（今読んでいる範囲では評価通りに最善を選べている）は、
  **「もっと深く読めば勝てたか」を測っていない**。強さ目的では探索の深さ/幅が議題に戻る。
- フェア化（相手手札透視を外す）は変な手を直さなかったが、これは体感の話で、強さの天井とは別軸。

## 3. 方針

**「ノイズ帯を破る測定 → 探索か評価かを 1 実験で切り分け → 勝った側に集中投資」。**
当て推量で置換表や ISMCTS に飛びつかず、まず限界要因を実測で確定してから投資する。

## 4. Phase 0 — 測定の分散低減 ★最優先・全ての前提

`cpu_arena` の A/B は現状 24〜30 局で、既知の **±35 Elo ノイズ帯**に埋もれる（過去 `+47Elo` 等を
自己評価で「ノイズ帯」と判定している）。強さを登るには、まず**少局数で有意差を出せる計測器**が要る。

- **共通乱数（common random numbers）/ 席固定ペアリング / antithetic seeds** で対戦間の分散を削る。
- **合格ゲート**: 同一条件 self-play で勝率の信頼区間がノイズ帯より十分狭いこと。以降すべての改善の
  合否判定器になる。

## 5. Phase 1 — 切り分け実験（探索 vs 評価）

Phase0 強化後の有意局数で、「**置換表(TT)＋ビーム拡張(4→6-8)＋horizon 延長**」をまとめて A/B する。

- **Elo が上がる** → 限界は探索 → **Phase 2A（探索路線）**へ。
- **上がらない/下がる** → 限界は評価（horizon amplification）→ **Phase 2B（評価路線）**へ。

この 1 実験で投資先を確定する。

## 6. Phase 2A — 探索路線（Phase1 が「探索限界」を示した時）

- **置換表(TT)**: 局面ハッシュ（Zobrist 相当）で同一局面の再評価を回避。`cpu_ai.py` のコメントに
  **「ビーム拡張 4→6-8 の前提」と明記された未実装項**。同レイテンシで深さ/幅を伸ばす最有力レバー。
- **反復深化＋壁時計デッドライン化**: 固定予算（`HARD_PER_MOVE_BUDGET`）を時間ベースへ。局面難度に
  応じて時間配分し、PV order（実装済み）と相乗。
- 各段 `cpu_arena`（Phase0 強化版）で Elo 非退行＋向上を確認。

## 7. Phase 2B — 評価路線（Phase1 が「評価限界」を示した時）

線形学習の不発（fair hard 320局・val acc 0.645）は**モデル容量の問題**＝非線形へ。

- **学習値の容量増**: GBDT / 小 MLP（stdlib 制約下で要設計）で winprob 予測精度を上げ直す。
- **または Elo ゲート付き手調整**: 学習重みを診断に使い、過大評価項を 1 項ずつ是正（各項を
  `cpu_arena` で Elo 判定）。
- winprob ブレンド配線（`OPCG_VALUE_BLEND_HARD`・α=0 で従来同値）は既存資産を流用。

## 8. Phase 3 — 不完全情報の正攻法（最終伸びしろ）

**ISMCTS / 決定化（determinization）**。隠れ情報（相手手札・裏ライフ・山札順）を複数サンプリングし、
各々を完全情報として探索 → 結果を平均して手を選ぶ。チートせず人間的な読み（ブラフ/受け）が出せる。
既存 `cpu_mcts`＋相手プロファイル（`cpu_opponent_model`）が土台。Phase2 で探索/評価の基礎を固めてから着手。

## 9. Phase 4 — 品質ゲート & SPEC 吸収

- 全テスト pass（`OPCG_LOG_SILENT=1 python -m pytest tests/ -q -s -p no:cacheprovider`）。
- 構造監査 0（`python tests/full_card_audit.py`）。
- 挙動を意図的に変えた箇所は `full_card_audit.py --regen` でベースライン再生成＋差分レビュー。
  検証済みデッキの挙動変化は `tests/test_verified_decks.py` に追記。
- 確定内容を `docs/SPEC.md §2.5` に吸収・`TEST_SPEC.md`（§3.1/§3.2）を更新。各 Phase の計測は
  `docs/reports/` に日付付きで追記（改変禁止）。

## 10. 依存関係・マイルストーン

```
Phase0 測定の分散低減 ─→ Phase1 切り分け実験 ─┬→ Phase2A 探索路線 ─┐
                                              └→ Phase2B 評価路線 ─┴→ Phase3 ISMCTS ─→ Phase4 ゲート
```

- **M0** Phase0 完了＝ノイズ帯を破る計測器が確定 ／ **M1** Phase1 完了＝限界要因（探索/評価）を特定 ／
  **M2** Phase2 完了＝特定した側で Elo 有意向上 ／ **M3** Phase3 完了＝不完全情報の正攻法導入。

## 11. 主なリスク

- アリーナの分散（±35 Elo のノイズ帯）→ Phase0 で先に潰す（順序厳守）。
- horizon amplification → 探索/評価を切り分けてから投資（Phase1）。深い探索が評価バイアスを増幅して
  退行する場合は評価路線を優先。
- 学習値のレイテンシ → `OPCG_VALUE_BLEND_HARD` の段階導入＋PyPy ワーカー経路で吸収。
- TT のハッシュ衝突/状態同一性 → clone 等価性テスト（`tests/test_cpu_make_unmake.py` 系）で機械照合。
