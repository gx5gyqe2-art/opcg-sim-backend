# ネット vocab 索引ズレ事故と恒久対策（2026-07-15 スナップショット）

v6 統合スモーク開始直後に生成ワーカー w1 が決定的クラッシュ（`IndexError: index 2660 is out of
bounds for axis 0 with size 2653`）。調査の結果、**カードDB更新に伴う学習済みネットの索引ズレ**という
serve/生成共通の構造欠陥と判明し、恒久対策（ネット付属 vocab）を実装した。

## 1. 事象と根本原因

- カードDB更新 `2c7457f`（update cards・2684枚）で新カード32枚（P-150/151・ST31〜36）が追加された。
- 符号化の `build_vocab` は **card_id ソート**で idx を振るため、新カードがソートの**途中**に挿入され
  （P-150 は PRB01 の前）、**既存371枚（PRB01〜ST30 全部）の idx が +2 ズレた**。
- 学習済みネット（gen2〜gen5・Emb 2653行＝2652枚時代）を現行 vocab で使うと:
  1. **PRB/ST カードは隣のカードの埋め込みを引く**（無言の品質破壊・serve でも発生）
  2. **新カード（idx≥2653）は範囲外参照でクラッシュ**（w1 の事象。生成は全リーダー回転で
     新カードがデッキに入るため即死。serve も新弾デッキで同様に落ちる状態だった)

## 2. 影響範囲の実測

- **本セッションの過去計測は無影響と確認**: v5採用の100局 arena・mark_gate/深探索プローブの
  fixtures はいずれも OP/EB 系デッキ（ズレの起点 PRB01 より前）のみで構成
  （arena 既定デッキ51種に PRB/ST/新弾カードが含まれないことを実測確認）。
- v5 run 本体は DB 更新**前**のクローンで生成・学習しており整合（ズレ発生は更新コミット以降のみ）。
- 実害が出たのは (a) v6 スモーク生成（クラッシュ・データ0件＝汚染なし）、
  (b) DB更新以降の serve での PRB/ST/新弾デッキ（品質劣化 or クラッシュ）。

## 3. 恒久対策（ネット付属 vocab・実装済み）

「符号化の対応表は**ネットが持つ**」に原則を変更（w1 提案の (A) 末尾ゼロ埋めは途中挿入ズレを直せず、
(C) DBピン留めは新カード対応が永久にできないため不採用）:

1. **`ValueNet.vocab_ids`**（訓練時の card_id 列・idx順）を npz に保存/ロード。
   複製系（expanded/to_v3/widened＝温スタート経路）も伝搬。
2. **serve**（`LearnedEngine`）: ネット付属 vocab で符号化（`encoder.vocab_from_ids`）。
   ネットが知らない新カードは既存の `_vidx` の `vocab.get(…, PAD)` により **UNK=0**＝
   ズレなし・範囲外なし（新カードは「見えない」だけ＝安全側）。
3. **学習**（`value_net.extend_to_vocab`・`p3_run.load_nets` が全経路で適用）: 現行DBとの差分を
   **末尾追記**で拡張（既存 idx 不変＝既存盤面の出力恒等をテストで固定）。新カードの Emb 行は
   ゼロ初期化（学習で育つ）、EffF 行は効果ASTから決定的に補充。生成ワーカーは拡張せず
   ネット付属 vocab をそのまま使う。
4. **同梱 gen2〜gen5 に訓練時DB（2652枚・`2c7457f~1`）の vocab を焼き込み**再保存
   （重み bit 不変を確認済み）。vocab_ids 無し＋行数不一致の npz は明示エラー（黙って進まない）。
5. 回帰テスト `tests/test_net_vocab_pinning.py`（**必須**・クラッシュ/誤評価の回帰を直接見張る）:
   同梱ネットの vocab_ids 保持／エンジンの訓練時 idx 復元（PRB01-001=2282）／新カード UNK／
   拡張の append-only・出力恒等／save-load 往復。

## 4. 運用メモ

- v6-net 種は焼き込み済み gen5 で**再シード**（旧種は vocab_ids 無しのため学習側で明示エラーになる）。
- ワーカー環境に numpy が未導入で初回起動が落ちた件（w1報告）: セッション初期化での
  `pip install numpy` が必要（コンテナイメージ依存・リポジトリの依存宣言整備は別課題）。
- 教訓: 「カードDBの更新」は**学習済みネットとの互換イベント**である。今後は vocab_ids が
  構造的に吸収するため運用手順は不要（新カードを学習させたい場合は次 run の extend で自動）。
