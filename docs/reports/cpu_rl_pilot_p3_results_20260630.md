## 0. ヘッドライン

**「学習RLは手作りCPUを超えられるか」を、計算資源ゼロの環境で直接対戦により実証した。**

| 段階 | vs 製品L1+α-β+PIMC4 | 
|---|---|
| Gen0（出発点・SL値＋uniform policy） | **0.450**（ほぼ互角） |
| **Gen2（自己対戦2世代・value＋policy）** | **0.925（圧勝・N=40・CI[0.801,0.974]）** |

さらに **Gen2 は製品CPUより約5倍速い**（sims160=323ms vs L1 pimc4=1656ms/手）。
＝**Rust移植なし・Python/numpyのまま、強い設定で配線してもレイテンシ予算内**。

## 1. 本走（自己対戦2世代・各10,000局）

- **Gen0**＝P2のSL価値net（cpu自己対戦7505局面で訓練）＋uniform policy。
- **Gen1**＝Gen0で自己対戦1万局→online学習（value+policy）。policy CE 1.42→1.10。
- **Gen2**＝Gen1で自己対戦1万局→online学習。policy CE 1.11→1.05。
- 各世代 = 4コア並列・sims=40・シャード60局・net-only checkpoint を専用ブランチへ force-push。
- **回収耐性が実地で機能**：コンテナ再起動＋二重起動事故が起きたが、
  net-only checkpoint（<2MB）＋自前カウンタの冪等性で、損失は直近1シャードのみ・自動収束で復旧。

## 2. 世代間クロス評価ゲート（損切り判定・N=400 CRN・Wilson CI）

| 比較 | 勝率 | 95%CI | 閾値 | 判定 | 性質 |
|---|---|---|---|---|---|
| Gen1 vs Gen0 | 0.900 | [0.867,0.926] | 0.55 | GO | policy獲得の底上げ込み |
| **Gen2 vs Gen1** | **0.568** | [0.519,0.615] | 0.51 | **GO** | フェア・有意・逓減域 |

- **2世代連続で統計的に有意な改善＝「自己対戦RLは伸び続ける」を陽性確認**。
- ただし世代改善は 0.90→0.568 と急速に逓減＝天井が近づく兆候もある（Gen3で天井位置を測れる・未実施）。
- 注意: N=50ピークは Gen2 vs Gen1=0.720 と上振れ。真値0.568（平均回帰）＝**損切り判定はN=400必須**の実証。

## 3. 直接対戦（参考・製品CPU比較）

- **Gen2+MCTS(sims160) vs 製品L1+α-β(pimc4)＝0.925**（37勝3敗・N=40・CI[0.801,0.974]）。
- P2の必要条件チェック（Gen0 vs L1=0.450）と同条件＝2世代のRLで **0.450→0.925**。
- 注意（water-oil）: α-β評価器 vs NN-MCTS の比較軸はレビュアーが「完全にはフェアでない」と指摘した点。
  ただしP2と同方法論で推移は一貫。N=40でCI幅はあるが下限0.801≫0.50で結論は堅い。

## 4. レイテンシ実測と配線可否（1手あたり・中盤局面）

| エージェント | ms/手 |
|---|---|
| Gen2 MCTS sims=100 | 193 |
| **Gen2 MCTS sims=160** | **323** |
| Gen2 MCTS sims=240 | 490 |
| L1 decide pimc=1 | 355 |
| **L1 decide pimc=4（本番）** | **1656** |

- **Gen2 sims160（0.925で勝つ設定）は製品L1(pimc4)より約5倍速い**。NN推論は軽く（~31µs/局面）、
  L1のPIMC4先読みの方が重かった＝**「強く繋ぐならRust前提」は覆った**。
- ユーザ許容「1.5×まで」に対し、Gen2は**製品より速い**＝**Python/numpyのまま初期配線してもレイテンシ予算を満たす**。
  sims240（490ms）に上げてさらに強くしても本番予算内。

### 配線の実像
- 製品CPUの呼び出し口＝`opcg_sim/api/app.py` の `/cpu/step`（`CPU_GAMES` にゲーム毎mem保持）→ `cpu_ai.decide/plan_turn`。
- 差し込みは decide レベルで「L1+α-β」の代わりに「Gen2+MCTS」へ分岐（返す move 辞書の形式は同一＝ドロップイン可能）。
- 必要作業（研究ハーネスの製品化）:
  - `tests/` の学習系モジュール（`rl_encoder`/`rl_net`/`az_policy`/`az_mcts_tree`/`opcg_game`/`opcg_action`）を `opcg_sim/src` へ移設。
  - net重み（`gen2_value.npz`/`gen2_policy.npz`・計~1.5MB）を同梱し起動時ロード。
  - `difficulty="learned"` 等のルート追加。
- ＝**根本的障壁はなく、エンジニアリング作業**。本走の学習資産（checkpoint）はそのまま使える。

## 5. 到達サマリ

| 段階 | 状態 |
|---|---|
| GATE A（ループ実装正しさ） | PASS |
| GATE B（探索健全＋PIMC） | PASS |
| P2（value→プレイ強度転換） | 陽性（Gen0≈互角） |
| P3 Gen1/Gen2（伸び続けるか） | **陽性（2世代連続で有意改善）** |
| **直接対戦（製品CPU超え）** | **陽性（0.925・かつ製品より高速）** |

**結論**：計算資源ゼロの環境で、学習型CPUが手作り型（磨き切って天井確認済み）を**明確に超え、かつ本番レイテンシ予算内に収まる**ことを実証。本格スケール（Rust+GPU）を待たずとも、**Python/numpyのGen2をそのまま配線するだけで、製品CPUを大幅に上回るCPUが得られる**見込み。

## 6. 未着手・次の選択肢

- **Gen3**：天井位置の測定（逓減カーブをもう1点・~6h）。持続性は既に示せたので必須ではない。
- **配線（製品化）**：§4の移設作業＋`difficulty="learned"`ルート。レイテンシ的には可能。要ユーザ判断・別PR。
- **スケール**（容量↑/sims↑/データ↑・Rust+GPU）：天井をさらに上げる本格投資。GO材料は本走で揃った。

## 7. 成果物（本走で使用・再利用可）

- 学習: `rl_encoder`/`rl_net`(value)/`az_policy`(pointer policy)/`opcg_action`(アクション符号化)
