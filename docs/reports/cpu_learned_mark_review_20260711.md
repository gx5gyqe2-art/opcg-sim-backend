## 2. 根因（4系統・いずれも個別カード非依存）

### F1: root 読み出しが argmax(N)＝「訪問の貼り付き」が Q の逆転を無視する

PUCT の訪問は prior と探索序盤の Q に貼り付く。探索後半に Q で逆転した代替
（@12: ATTACH_DON が Q で +0.084 上）があっても、160 sims では訪問数が追い付かず
argmax(N) は劣後手を返す。@12・@24 は**探索自体は正解を見つけていた**（Q 順位は人間指摘と一致）
のに読み出しで捨てていた＝純粋な読み出し問題。

**実装（一般解）**: 読み出しを「最多訪問グループ基準、ただし十分訪問（n ≥ 0.2·n_top）の代替の
LCB = q − z/√n が上回れば乗り換え」へ（`cpu_learned._select_root_group`・z=1.0）。低訪問 Q の
楽観ノイズは 1/√n 悲観補正と訪問率ゲートで抑える（Leela/KataGo の LCB move selection と同族）。
探索・訪問統計・トレースは不変＝読み出しのみ。z=0 で従来挙動に一致（ロールバック容易）。

記録統計への適用（決定論的検証・`tests/test_learned_root_readout.py` に固定化）:
- @12: ATTACK → **ATTACH_DON(リーダー)** ✅ 指摘どおり
- @24: ATTACK(リーダー) → **ATTACK(OP15-105 波)** ✅ 指摘の筋（波を絡める）へ
- 他6件: 変化なし（悪化もなし）。@49/@63 は Q 自体が誤っているため読み出しでは直らない→F2/F3/F4。

### F2: 終局値が深さ無差別（±1固定）＝敗勢で全候補 −1 に飽和し防御が崩壊する

@63 は全候補 q=−1.000（決定化した世界で何をしても負け）。この時 MCTS は prior 任せになり
PASS を選ぶ＝「ライフの価値が高い局面で守らない」。L1 は ±(W_WIN − ply) で最短リーサル/最長抵抗を
選ぶが、TreeMCTS の terminal は ±1 固定でこの構造を持っていなかった。

**実装（一般解）**: terminal backup を ±max(TERM_FLOOR=0.5, 1 − TERM_DECAY(=0.02)·depth) に
（`mcts.TreeMCTS._term_scale`・非終局の葉価値は素通し）。「勝ちは近い方・負けは遠い方」が
Q に距離として現れ、負け確定の世界でもカウンターで1手粘る側が PASS より高くなる。
ユーザ指摘の但し書き（相手ライフが少なくゲームが短いなら受けることもある）も、
「受けて勝ちに向かう線があるなら terminal 符号が + になる」ため同じ機構で自然に扱われる。
`tests/test_mcts_terminal_decay.py`（汎用グラフゲーム）で「遅い負けを選ぶ／速い勝ちを選ぶ／
床で下げ止まる」を固定化。

### F3: decide ごとの PIMC 世界線引き直し＝ターン内の計画非一貫（無駄ドンの主因）

learned の decide は毎手、相手伏せ手札を1世界にサンプルして探索する。**ターン内の連続 decide で
世界を引き直す**ため、「付与を正当化した攻撃プラン」が次の decide の別世界で棄却される:
- @47-48 で菊之丞に2ドン付与 → @49 の新世界では ATTACK q=−0.009 → 付与だけ残して TURN_END（指摘7）。
- @28 付与（その世界では攻撃 q=+0.145）→ @29 の新世界で攻撃 q=−0.611 に崩落、しかしドンは支払い済み（指摘4・5）。

**実装（一般解）**: 同一 (game, turn, player) の間は決定化 seed を固定（`LearnedEngine._world_rng`・
`SERVE_STICKY_WORLD=True`）。ターン内の全 decide が同一世界でプランを評価する＝「付与→攻撃」が
同じ前提で完結する。serve 専用（自己対戦・学習データ生成は不変）。seed は初回 decide の rng から
引くため global random の消費量は従来と同一＝リプレイ決定論を保つ（`test_cpu_learned.py` の
決定論テストで担保）。公開情報の更新（相手がカウンターを切った等）は盤面側から反映され続ける。

### F4: value net の評価誤差（ドン付与まわりの系統誤差）— 本 PR では未修正・学習側の宿題

読み出し・探索を直しても Q 自体が誤っているケースが残る:
- @16: 攻撃後の付与（純損）に q=+0.027 > TURN_END −0.174。net が「付与ドン」を静的資産として過大評価。
- @43: リーダー付与が q=−0.597（指摘は付与推奨）。逆方向にも誤る＝付与特徴の学習が不十分。
- @49: 2ドン載せた菊之丞の攻撃 q=−0.009 vs TURN_END 0.42 の妥当性は世界依存で、netの過小評価が疑わしい。

一般解は学習側: `gen3_adoption_20260711.md` §4 の結論（勝敗ラベル単独では忘却が起きる→
**教師値×実戦勝敗の混合損失 or MCTS改善値ラベル**）に、本マークのような
**人間マーク回帰をゲート指標に含めた次世代学習**として引き継ぐ。今回の3実装（F1-F3）は
その学習が進むまでの間も**net を差し替えるだけで一緒に効き続ける**探索層の恒久改善。

## 3. 検証

  `tests/test_mcts_terminal_decay.py`（汎用グラフゲームで減衰の性質・4件）。いずれも `cpu_infra`。
- **マーク回帰（読み出し）**: 記録統計に新読み出しを適用し @12/@24 が指摘どおり flip・他6件は不変
  （§2 F1）。盤面復元での end-to-end 再評価は今回の記録が frames 無しの replay 出力だったため不可
  （全手注入の再生も step5 の山札覗き効果で乱数ドリフト＝`replay_reeval.py` docstring の既知の限界を
  追認）。frames は `GET /api/game/{id}/replay/frames` に存在する＝**マーク採取時はこちらを保存**
  すれば `replay_reeval.py` の before/after 回帰がそのまま使える。
- **アリーナ健全性**: learned(本変更) vs L1-hard・CRN 対照ペア（結果は §4）。
- **品質ゲート**: `make test` green（push 前実行）。

## 4. アリーナ結果（learned vs L1 hard・対照ペア12＝24局・CRN 同一シード seed0=4242）

| | 勝率 | Elo（ペア単位95%CI） | s/局 |
|---|---|---|---|
| before（HEAD 4569d99） | 0.417 | −58 [−263, +110] | 12.9 |
| after（本変更 F1+F2+F3） | 0.417 | −58 [−226, +85] | 12.9 |

**総勝率は完全一致・レイテンシ不変**（ペア内訳のみ僅差＝一部の局で手が変わったが勝敗合計は同じ）。
大幅劣化がないことの健全性確認が目的で、N が小さいため Elo の点推定で採否を語らない
（`gen3_adoption_20260711.md` §2 と同じ流儀）。本変更の狙いは Elo でなく
**人間から見た「変な手」（Q劣後手・無駄ドン・無抵抗）の削減**であり、その回帰はマーク統計（§2 F1）と
単体テスト2件で固定した。強度への影響を厳密に測る場合は多様97ハーネス（`p3_vs_l1.py`）で
before/after を回す（次回マーク蓄積時にまとめてで可）。
