# 学習eval 大規模化スパイク 設計（2026-06-29・別プロジェクト規模の本丸・可否判定用）

> **種別: 報告（点）**。設計スナップショット。前提＝`cpu_strength_plan_20260628.md` §K（現アーキは上限近傍）
> ＋`cpu_correctness_instruments_20260628.md`（客観計器も「CPUは健全」を裏付け）。
> 結論：**測れる精度は頭打ち。質的飛躍＝学習RL（AlphaZero型）が唯一の本命**。本書はその**最小デリスク**＝
> 「**学習価値をデータ規模で作れば手作りL1を超えるか**」を安く決着させるスパイクの設計。

## 0. なぜこのスパイクか（唯一未検証の本丸仮説）
過去の学習価値は **極小ブレンド×極小データ（2110行・val_acc 0.645）→ Elo中立/−70** で撤去した。だが
`cpu_phase3a_fair_value_relearn` 報告自身が「**データを綺麗にしたら val_acc 改善・確定には大規模データが要る
（未実施）**」と明記。**＝"本物"は一度も試していない**。AlphaZero型フルRL（方策ネット＋ネット誘導MCTS＋
反復自己対戦）に踏み込む前に、**最小の問い「価値ネットは規模で L1 を超えるか」だけを先に潰す**。

## 1. スパイクの問い（Go/No-Go ゲート）
> **大規模・outcome ラベルで学習した価値ネットを、既存 PIMC 枠の葉評価として L1 と差し替え、
> 本番構成(pimc=4)の Elo A/B で L1 に有意勝ちするか？**
- **勝つ → GO**：フルRL（方策ネット＋自己対戦反復＋必要ならRustコア）へ投資する価値あり。
- **中立/負け → NO-GO**：学習evalは規模でも効かない＝本命の前提が偽＝**数ヶ月を節約して棄却**。

## 2. 設計

### 2.1 データ生成（規模が肝）
- **教師＝現行の強い PIMC CPU の自己対戦**（fair・実デッキ deckgen で分布多様化）。
- **ラベル＝局面の最終勝敗**（AlphaZero の value ターゲット＝バイアスなし・高分散）。ターン境界スナップショット。
- **規模＝過去の 10〜100倍（目標 ~10^5〜10^6 局面）**。純Pythonは遅いので **PyPy＋multiprocessing 並列**で数日。
  （この段階では **Rust は使わない**＝スパイクを安く保つ。データ生成が律速と分かってから併走で検討。）
- **データ多様化**：荒らしエージェント混入＋全リーダープール（過去 `db29dce` の知見）で state 被覆を広げる。

### 2.2 入力表現（**過去失敗の最有力原因＝ここ**）
> **致命的論点**：ネットの入力が「L1 が計算する特徴量」と同じなら、ネットは**せいぜい L1 の関数を再現するだけ
> ＝L1 を超えられない**。過去の不発はこれの疑いが濃い。**L1 を超えるには L1 より生・豊富な入力**が要る。
- **採用＝半生表現**：盤面を構造化テンソル/多チャネルで（各キャラの cost/power/keyword/rest/attached_don、
  ライフ枚数、ドン、手札枚数、リーダー、相手公開情報…）＝L1 が捨てている相互作用を学べる粒度。
- 容量はデータ規模に合わせる（過去の極小MLPでなく適正サイズMLP/小CNN）。推論は配信で許容できる軽さに。

### 2.3 評価ゲート（A/B）
- 学習netを **PIMC 枠の葉評価**として差し替え（or L1とブレンド α掃引）、**本番構成(pimc=4)で L1 とElo A/B**。
- 計測ゲート（憲法）：**CRN＋席交互・200→500ペア・95%CI が 0 を跨がない**。既存 `arena_parallel` を使用。
- 副指標：val_acc／キャリブレーション（Phase3a は綺麗データで val_acc 改善を観測）。**ただし val改善≠Elo**
  （過去 −70Elo の前科）＝**最終判定は必ず Elo**。
- **データ規模カーブ**：N を 10倍刻みで増やし **val_acc と Elo が N で伸び続けるか**を見る（伸び続け＝規模が
  効いている／飽和＝規模では超えられない）。

## 3. スコープ境界（スパイクを安く保つ）
- **やらない**：方策ネット・ネット誘導MCTS・自己対戦反復(RL)・Rustコア。＝フルRLは GO 後。
- **やる**：大規模データ生成 ＋ 半生表現の価値ネット教師あり学習 ＋ L1との Elo A/B ＋ 規模カーブ。
- 教師は**現行CPUの自己対戦**（RL反復はしない）。「規模で L1 を超えるか」だけを問う最小実験。

## 4. 判定基準
- **GO**：本番構成で net-eval が L1 に **有意 +Elo**（95%CI>0）かつ規模カーブが上り。
- **NO-GO**：規模を上げても中立/負け or 飽和＝学習evalは効かない＝本命棄却。

## 5. 主要リスク（レビューで潰す）
1. **入力表現の天井**：特徴がL1相当だとネットはL1を超えられない（§2.2）。半生表現の設計が成否を分ける。
2. **outcome ラベルの分散**：高分散→大データ必要。深い探索value(eval偏り) vs outcome(不偏) のどちらを target にするか。
3. **データ規模の現実性**：純Python/PyPy で 10^5〜10^6 局面が数日で出るか。出ないなら Rust 前倒しの是非。
4. **自己対戦データのバイアス**：現行CPUが到達しない state（人間が作る局面）を被覆できるか。
5. **鏡均衡/最終検証**：自己対戦Eloで GO 判定して良いか、固定/人間相手の検証を GO 前に挟むべきか。
6. **配信レイテンシ**：net推論を1秒予算×pimc4 に載せられる軽さか（GO後の制約を先に見積もる）。

## 6. 着手順
1. データ生成基盤（PyPy並列・outcomeラベル・荒らし混入・規模可変）。
2. 半生表現＋適正サイズ価値ネットの学習（規模カーブ）。
3. PIMC枠への差し替え＋本番構成 Elo A/B（CRN・500ペア）。
4. 判定 → GO なら フルRL設計へ／NO-GO なら本命棄却を記録。

## 追補（2026-06-29・外部AIレビュー5往復で確定した改訂版＝§1-6 を上書き）

> 上の §1-6（学習価値を**α-βの葉に差してL1とElo比較**する版）は**棄却**。レビューで3つの致命傷が判明：
> ①**α-β+NN は water-oil**（minimax が NN の幻覚＝外れ値を能動的に極大化＝偽陰性ゲート。AlphaZeroが
> MCTS を採る理由）。②**模倣の天井**（教師あり×現行CPU outcome は教師の方策歪みを蒸留＝教師超え不可）。
> ③**入力表現の天井**（L1相当特徴ではL1の関数再現が上限）。よって「安い教師ありスパイク」は**存在しない**。

### A. 確定した最小の有効スパイク＝「Python先行・小型AlphaZero縦切り」
- **Dual-Net**（value＋policy 2ヘッド）。**入力＝半生表現＋カードID Embedding**（カード固有シナジーを学べる粒度）。
- **世代0＝教師あり事前学習**（現行CPU自己対戦＋**ノイズ注入**で局所最適から散らす）＝重みを常識的初期値へ。
- **世代1〜3＝最小RLループ**（Dual-Net誘導の**最小MCTS**で自己対戦→新データ→再学習を**3反復**）。
  ＝policy improvement を最小限入れて**「教師超え」を機能証明**する（教師ありだけでは偽陰性）。
- **評価＝α-β禁止・自己対戦Elo禁止**。①MCTSはNN幻覚を平均化で吸う②自己対戦Eloは鏡均衡。
- **Rust は GO 後**：実測で移植コア ~9,000 LOC（効果実行 ~5,300＝最難）・2,652カードへ bit-exact＝**~9-12+人月**。
  これを **GO 判定前**に払うのは経済学的に転倒（縦切りの鮮度要件＝1世代~数日×3＝**~2週、コストはクラウド$**）。
  ＝**スパイクは Python(クラウド並列)で組み、Rust は GO ゲート後の本番スケール化に回す**。

### B. GO/NO-GO ゲート（自己対戦Elo を完全排除）
1. **定量（対L1・非鏡クロス評価）**：MCTS側に**本番想定の固定シミュレーション数**（例 800/手）、L1(α-β)側に
   **フル予算**を与える非対称規約で、CRN・**500ペア・勝率>55%（95%CI>0）**。
2. **定性（L1の死角の破壊）**：①で構築した **Lethal/Survive Puzzle Suite** のうち **L1 が Regret>0 で外した盤面**を
   RL+MCTS に食わせ、**オラクル一致手を拾えるか**。L1 が構造的に選べなかった手を拾えれば質的飛躍の証拠＝GO。
   - 補足：パズル Top-1 一致（教師戦術の模倣）は**床（sanity）**であって GO 信号ではない（現行CPUは簡単リーサル
     を既に Regret 0 で解く＝一致は「L1に並んだ」証拠で「超えた」証拠でない）。GO は上記2ゲートで判定。
3. **判定**：3世代で対L1勝率が SL時点から**上昇トレンド＋55%突破**かつ定性ゲート通過＝**GO**（→Rust＋本番RL）。
   3世代回しても 50% を割れない＝**NO-GO**（本命棄却・数ヶ月を節約）。

### C. 実装上の制約（本リポジトリ環境）
- 本環境＝**numpy/pypy3・4コア・torch/GPU 無し**。よって**土台コードの構築と小規模検証はここで可能**だが、
  **GO/NO-GO 本走（大規模データ＋NN学習×3世代）は外部計算資源が必須**（クラウドコア＋torch/GPU）。
- ローカル足場のネットは **numpy 実装**（旧 `train_mlp` と同型＝numpy学習＋pure-Python/JSON推論）で組み、
  本走時に torch へ差し替え可能にする。

### D. 改訂版・着手順（Python先行）
1. **盤面エンコーダ**（GameManager状態 → 半生特徴ベクトル＋カードIDインデックス）。決定的・単体テスト可・**ここで構築/検証**。
2. **データ生成ハーネス**（自己対戦＋ノイズ注入＋outcomeラベル → エンコード済み局面）。小規模検証は**ここで**、本走は外部。
3. **最小MCTS**（純Python・Dual-Net 誘導）。自明局面で正しく解くことを**ここで検証**。
4. **Dual-Net**（numpy・value+policy・カードEmbedding）。tiny集合を過学習できることを**ここで検証**。
5. **3世代RLループ＋クロス評価ゲート**（外部計算資源で本走）。
6. 判定 → GO なら Rust＋本番RL／NO-GO なら本命棄却を記録。

> 本追補が確定設計。次の実装は **D-1 盤面エンコーダ**から（決定的・テスト可・環境内で完結）。

## 追補2（2026-06-29・初回スケールカーブ結果＝計算資源無し環境での事前フィルタ）

計算資源が無いため、GO/NO-GO 本走（RL+MCTS+Elo・大規模）の代わりに**核心仮説の事前フィルタ**＝
「学習評価器は規模で L1 を超えるか」を **held-out 勝者予測精度（sign-accuracy）のスケールカーブ**で測った。

- データ：現行CPU(教師)＋ノイズ注入(eps=0.3)の自己対戦 **480ゲーム→25,686局面**（4並列・numpy）。
- 評価：**game単位 split**（局面リーク防止＝局面単位splitは net を過大評価する。実測でリーク版0.762→正版0.540）。
- net＝numpy value net（半生特徴＋カードID Embedding・hidden128/d_emb24）、N を増やして再学習。

| N(train局面) | net sign-acc | vs L1(0.581) |
|---|---|---|
| 1,000 | 0.586 | +0.004 |
| 3,000 | 0.607 | +0.026 |
| 10,000 | 0.619 | +0.038 |
| 22,623 | 0.629 | **+0.047** |

→ **net は L1 を超え・N で単調増加・22kで未飽和**＝「規模が効く」方向の**肯定的事前信号**（本セッション初の前向き結果）。

### 但し書き（過大評価しない）
- **sign-acc ≠ Elo**：評価器が正確でも α-β 葉に差すと幻覚搾取で負ける前科（−70Elo）。**最終GOは MCTS×Elo（要計算資源）**。
- 優位は +0.047 と小さいが、**単調増加・gap拡大・未飽和のトレンド**が信号。held-out 58ゲーム＝単点ノイズは大。
- **全局面が同一合成デッキ**＝カードEmbeddingがほぼ遊休。**多様デッキ（deckgen）なら更に伸びうる**＝過小評価の可能性。
- 結論：**これは「本命を諦めるな」の事前信号**。次は (a) 規模拡張(10^5)でトレンド確認、(b) 多様デッキ、
  (c) 計算資源が出来たら MCTS×Elo 本走。NO-GO ではない＝フルRLへ進む価値が定量的に裏付いた。
