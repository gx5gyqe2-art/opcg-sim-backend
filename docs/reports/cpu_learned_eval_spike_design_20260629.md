# 学習eval 大規模化スパイク 設計（2026-06-29・別プロジェクト規模の本丸・可否判定用）

> **種別: 報告（点）**。設計スナップショット。前提＝`cpu_strength_plan_20260628.md` §K（現アーキは上限近傍）
> ＋`cpu_correctness_instruments_20260628.md`（客観計器も「CPUは健全」を裏付け）。
> 結論：**測れる精度は頭打ち。質的飛躍＝学習RL（AlphaZero型）が唯一の本命**。本書はその**最小デリスク**＝
> 「**学習価値をデータ規模で作れば手作りL1を超えるか**」を安く決着させるスパイクの設計。

## 0. なぜこのスパイクか（唯一未検証の本丸仮説）
過去の学習価値は **極小ブレンド×極小データ（2110行・val_acc 0.645）→ Elo中立/−70** で撤去した。だが
`cpu_phase3a_fair_value_relearn` 報告自身が「**データを綺麗にしたら val_acc 改善・確定には大規模データが要る
（未実施）**」と明記。**＝"本物"は一度も試していない**。AlphaZero型フルRL（方策ネット＋ネット誘導MCTS＋
反復自己対戦）に踏み込む前に、**最小の問い「価値ネットは規模で L1 を超えるか」だけを先に潰す**。

## 1. スパイクの問い（Go/No-Go ゲート）
> **大規模・outcome ラベルで学習した価値ネットを、既存 PIMC 枠の葉評価として L1 と差し替え、
> 本番構成(pimc=4)の Elo A/B で L1 に有意勝ちするか？**
- **勝つ → GO**：フルRL（方策ネット＋自己対戦反復＋必要ならRustコア）へ投資する価値あり。
- **中立/負け → NO-GO**：学習evalは規模でも効かない＝本命の前提が偽＝**数ヶ月を節約して棄却**。

## 2. 設計

### 2.1 データ生成（規模が肝）
- **教師＝現行の強い PIMC CPU の自己対戦**（fair・実デッキ deckgen で分布多様化）。
- **ラベル＝局面の最終勝敗**（AlphaZero の value ターゲット＝バイアスなし・高分散）。ターン境界スナップショット。
- **規模＝過去の 10〜100倍（目標 ~10^5〜10^6 局面）**。純Pythonは遅いので **PyPy＋multiprocessing 並列**で数日。
  （この段階では **Rust は使わない**＝スパイクを安く保つ。データ生成が律速と分かってから併走で検討。）
- **データ多様化**：荒らしエージェント混入＋全リーダープール（過去 `db29dce` の知見）で state 被覆を広げる。

### 2.2 入力表現（**過去失敗の最有力原因＝ここ**）
> **致命的論点**：ネットの入力が「L1 が計算する特徴量」と同じなら、ネットは**せいぜい L1 の関数を再現するだけ
> ＝L1 を超えられない**。過去の不発はこれの疑いが濃い。**L1 を超えるには L1 より生・豊富な入力**が要る。
- **採用＝半生表現**：盤面を構造化テンソル/多チャネルで（各キャラの cost/power/keyword/rest/attached_don、
  ライフ枚数、ドン、手札枚数、リーダー、相手公開情報…）＝L1 が捨てている相互作用を学べる粒度。
- 容量はデータ規模に合わせる（過去の極小MLPでなく適正サイズMLP/小CNN）。推論は配信で許容できる軽さに。

### 2.3 評価ゲート（A/B）
- 学習netを **PIMC 枠の葉評価**として差し替え（or L1とブレンド α掃引）、**本番構成(pimc=4)で L1 とElo A/B**。
- 計測ゲート（憲法）：**CRN＋席交互・200→500ペア・95%CI が 0 を跨がない**。既存 `arena_parallel` を使用。
- 副指標：val_acc／キャリブレーション（Phase3a は綺麗データで val_acc 改善を観測）。**ただし val改善≠Elo**
  （過去 −70Elo の前科）＝**最終判定は必ず Elo**。
- **データ規模カーブ**：N を 10倍刻みで増やし **val_acc と Elo が N で伸び続けるか**を見る（伸び続け＝規模が
  効いている／飽和＝規模では超えられない）。

## 3. スコープ境界（スパイクを安く保つ）
- **やらない**：方策ネット・ネット誘導MCTS・自己対戦反復(RL)・Rustコア。＝フルRLは GO 後。
- **やる**：大規模データ生成 ＋ 半生表現の価値ネット教師あり学習 ＋ L1との Elo A/B ＋ 規模カーブ。
- 教師は**現行CPUの自己対戦**（RL反復はしない）。「規模で L1 を超えるか」だけを問う最小実験。

## 4. 判定基準
- **GO**：本番構成で net-eval が L1 に **有意 +Elo**（95%CI>0）かつ規模カーブが上り。
- **NO-GO**：規模を上げても中立/負け or 飽和＝学習evalは効かない＝本命棄却。

## 5. 主要リスク（レビューで潰す）
1. **入力表現の天井**：特徴がL1相当だとネットはL1を超えられない（§2.2）。半生表現の設計が成否を分ける。
2. **outcome ラベルの分散**：高分散→大データ必要。深い探索value(eval偏り) vs outcome(不偏) のどちらを target にするか。
3. **データ規模の現実性**：純Python/PyPy で 10^5〜10^6 局面が数日で出るか。出ないなら Rust 前倒しの是非。
4. **自己対戦データのバイアス**：現行CPUが到達しない state（人間が作る局面）を被覆できるか。
5. **鏡均衡/最終検証**：自己対戦Eloで GO 判定して良いか、固定/人間相手の検証を GO 前に挟むべきか。
6. **配信レイテンシ**：net推論を1秒予算×pimc4 に載せられる軽さか（GO後の制約を先に見積もる）。

## 6. 着手順
1. データ生成基盤（PyPy並列・outcomeラベル・荒らし混入・規模可変）。
2. 半生表現＋適正サイズ価値ネットの学習（規模カーブ）。
3. PIMC枠への差し替え＋本番構成 Elo A/B（CRN・500ペア）。
4. 判定 → GO なら フルRL設計へ／NO-GO なら本命棄却を記録。
