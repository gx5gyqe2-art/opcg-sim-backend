## 3. 根因の統合仮説

### R1. 「守りすぎ」は防御温度延長の過剰注入ではない【調査完了・仮説否定】

当初「v4 の防御応答の温度延長が守り手を過剰注入し value を『守り＝善』へ振らせた」と疑ったが、
**`defense_rate_probe.py` の計測（24局・630防御局面）で否定された**:

| 系統 | 守り率 | 対 L1 |
|---|---|---|
| net argmax（素の意向） | 0.433 | **L1 より守らない**（−0.067） |
| 温度1期待（データ挙動） | 0.539 | +0.039（軽微な超過） |
| L1-hard（良質目安） | 0.500 | 基準 |

net の素の意向はむしろ**守らなさすぎ**（0.433<0.500）で、温度延長はデータを L1 とほぼ同水準
（0.539≈0.500）へ**補正している**＝過剰注入ではない。よって C1「守りすぎ」は**集計の体質ではなく
局面依存**で、内訳は @115＝**R3 負けq飽和**（詰み局面で「一番マシ」がカウンター）、@82＝優勢局面で
カウンターを +0.05 過大評価する**軽い較正ズレ**。

→ **方針変更**: 防御温度延長の「条件化」は**主レバーから降格**（原因でないため）。温度延長は 0.539 と
L1 を +0.04 だけ超過するので、気になれば軽い dial-back（守り価値の低い局面のみ温度を下げる）を
**任意の微調整**として残すが、v5 の主眼は R3（負けq飽和）と D1（付与）へ移す。

### R2. 較正の不安定（C2/C3）＝「程度」を教える信号の欠如

付与も攻撃も「打つ/打たない」の二値でなく**どれだけ価値があるか**の連続量が問われるが、勝敗±1＋
q_root ラベルは局面価値であって「この付与で戦闘結果がどれだけ改善するか」の**差分**を直接教えない。
→ v5 は §4-3 マーク局面シード＋§4-5 encoder 拡張（自デッキ残）でデータ・入力の両面から底上げし、
必要なら「付与前後の value 差分」報酬プローブを診断計器化（学習への直接投入は後段で検討）。

### R3. 負けq飽和（C4）＝残りターン補助の活用不足

value が敗勢で −1 に張り付くと手が区別できない。v4 で入れた**残りターン補助ヘッド**は「あと何ターンで
負けるか」を持つので、**敗勢でも「長く粘る手」を選べる材料**になる。現状 aux は表現学習の誘導のみで
手選択に使っていない。→ v5 で aux 重みの調整、または探索の葉評価に「aux 由来の粘り項」を薄く混ぜる
実験（終局深さ減衰 F2 の中盤版）。

## 4. v5 run の構成（v4 配管を流用・差分のみ）

v4 の生成/学習/ゲート/退避/keep-alive 運用は流用。差分:

0. **前提調査（R1）**: 済（`defense_rate_probe.py`）＝守りすぎは防御温度延長でない（§3-R1）。防御温度の
   条件化は**任意の微調整**に降格。以下は調査後の確定版レバー。
1. **負けq飽和の緩和（R3・C4＝守りすぎ@115/敗勢の無駄手の主因・最優先）**: v4 の残りターン補助ヘッドを
   手選択に効かせる。案(a) 探索の葉評価に「aux 由来の粘り項」を薄く混ぜる（終局深さ減衰 F2 の中盤版＝
   詰み局面で『長く粘る手 > 無意味な守り/攻撃』を区別）、案(b) value ラベルの飽和緩和（負け側も
   q_root の距離成分を残す）。まず案(a) を serve 側で A/B（学習不要で効果を先読み）。
2. **マーク局面シード（D1/C2 の最優先対策）**: v4 の失敗局面（C2/D1 の @64/@68・C5 の @3・
   過去 g1@12/@24 の付与正着・守りすぎ @82）を自己対戦の開始局面に混ぜ、**観測された失敗モードそのもの**
   を in-distribution 化。
3. **encoder 拡張（D3・C5・append-only）**: 自デッキ残の集約特徴（残カウンター密度・防御札残量・
   キーカード残枚数）を scalars 末尾に追加＝時計誤差の対面別偏りと資源評価（ライフvs手札）を底上げ。
   v4 からの温スタートは既存の恒等拡張機構（`expanded`/warm_start）で自動追従。
4. **忘却抑制（D2）**: (a) mark_gate/対v3 の同時後退を検知する**ピーク自動アラート**を監視サイクルへ
   組込む・(b) 学習損失に **v4 を教師にした軽量 KL 蒸留正則化**を加える試験（EWC より軽実装）。
5. **温スタート**: v4（`gen4_*.npz`）から。α/aux は v4 起点でホールドアウト較正。
6. **ゲート拡張**: 既存16マーク＋**本14マーク**を fixtures に追加（`g3_v4_replay_*`）。C2/D1 の付与較正・
   C3・C4（敗勢の無駄手）を必達に含める。対L1多様97 は v4 の 0.812 非劣。
7. **（任意）防御温度の軽 dial-back**: 温度延長のデータ守り率 0.539 を L1 の 0.50 へ寄せる微調整。主眼外。

### 4-1補. aux 粘り項（負けq飽和の緩和）【実装済み・2026-07-13】

§4-1 案(a) を serve 側で実装（学習不要）。葉評価が飽和域（|v| ≥ `AUX_SAT_START`=0.8）のとき、v4 の
残りターン補助ヘッドの予測 t̂ で振幅を減衰する: `v' = v·max(TERM_FLOOR, 1 − AUX_TIE_DECAY·t̂·sat)`、
`sat = clip((|v|−0.8)/0.2, 0, 1)`。終局深さ減衰（F2・TERM_DECAY）の「終局に届かない飽和葉」への拡張＝
**敗勢では延命する手（t̂ 大）を、優勢では速い勝ち（t̂ 小）を選好**。非飽和域は恒等＝中間域の較正は不変。
`SERVE_AUX_TIEBREAK`（`config.py`）でゲート・`LearnedEngine(aux_tiebreak=)` でエンジン別 A/B 可・serve 専用
（自己対戦データ生成は不変）。

検証: マーク局面 A/B（ON vs OFF）で決断フリップなし＝C4系正着（@115 無意味な守り→PASS・g1@63 粘る
カウンター→守る）を維持しつつ、飽和葉の q のみが延命方向へ動く（例 @38 q −0.29→+0.04・選択は不変）。
arena gen4(ON) vs gen4(OFF) = 13/24（0.542・CI[0.35,0.72]＝中立・退行なし・粘り項は趨勢が決した局面
のみ作用するので勝率自体は動かさない設計）。レイテンシ 303ms vs 302ms/手（forward 共有で不変）・
make test 1353 passed。**q_root 飽和の学習側緩和（案(b)）は本走の value ラベルで別途対処**（§4-1 案b・本計画の主眼）。

### 4-3補. encoder 拡張（自デッキ残の集約特徴）【実装済み・2026-07-13】

§4-3 を実装（符号化世代 **v4**・append-only）。v3（scalars 46）末尾に `me.deck`（自分の残ライブラリ）の
守り/資源集約 5 値を追加（51）: `[残カウンター総量, カウンター札密度, ブロッカー残, イベント残, 高コストキャラ残]`。
「自分の山札にどれだけ守り札/カウンターが残るか」を可視化＝薄いライフの precious 価値（C5）と
残ターン読み（D3・時計）を底上げする。**カード個別に依存しない汎用量**（counter/keyword/type の集計のみ・
特定カードIDのハードコード無し）。相手デッキは非公開ゆえ**自分のみ**（公平性契約＝隠れ情報を符号化しない）。

配管: `_SCALARS_BY_VERSION` に `4:51` を1行追加＋`encode` の `version>=4` 分岐のみ（版判定・温スタート・
ドリフト検知は既存機構が自動追従）。生成/学習/run の 3 CLI（`pd_gen`/`pd_learn`/`p3_run`）は `--enc-version 4`
を許可。value/policy とも既存の append-only 温スタート（`expanded`/`warm_start_*`）でそのまま拡張。

検証: **恒等温スタート必達**＝出荷 gen4（v3）を v4 へ拡張すると v4 符号化での value/aux 出力が数値的に完全一致
（Δ≈機械精度・新5スカラーが W1 のゼロ行に当たる）＝「データ再生成しても学習前は挙動不変」。集約値の
定義一致・自デッキのみ（相手改変で不変）・空デッキ安全を単体テストで固定（`test_rl_encoder_v4.py`）。
gen4→v4 温スタート net で自己対戦1局が enc_version=4 で完走（net feat_dim 131 = feature_dim(4)）。

> 本走の温スタート土台: 出荷 gen4（v3）→ v4 拡張ネットを checkpoint 枝に種として置いて起動する（§4-5）。
> `load_nets` は checkpoint があればそれを優先するので、cold-fallback（出荷 gen2 から v1→v4 温スタート）とは
> 別に、明示シードで gen4 の実力を引き継ぐ。

### 4-2補. マーク局面シード【実装済み・2026-07-13】

§4-2 を実装。人間が悪手をマークした実対局の**失敗局面**（MAIN 手番マーク）を静的フレームから復元し
（`tests/harness/mark_seeds.py`・replay_reeval の復元を流用・決定論）、自己対戦の開始局面プールにする。
`selfplay_game(seed_boards=, seed_frac=)` が各局を確率 `seed_frac` でプールから開始（残りは従来の
turn1 `new_game`）＝観測された失敗モードそのものを in-distribution 化する。中盤開始でも軌跡・ラベル
（勝敗/q_root/turns_left）は通常経路と同一（turns_left は終局ターンから逆算＝開始ターンに依らず正しい）。

配線: `pd_gen`/`p3_run` に `--mark-seed-frac`（既定 0＝OFF）・ワーカーは frac>0 のとき1回だけプール復元
（`_W["seed_boards"]`）。カウンター/戦闘マークは戦闘途中の再開が近似的なため除外＝MAIN 手番のみ
（現状 fixtures 3局から 26 局面）。**`seed_frac=0` は rng 消費順まで従来と完全一致**（seed 判定の乱数を
引かない）＝シード OFF の本走は v4 生成と bit 単位で同一データ。検証: `test_mark_seeds.py`
（プレイ可能・決定論・シード開始で完走・frac=0 の挙動不変ゲート）・ワーカー経路スモーク（frac=1.0 で
26 盤面ロード・生成完動）。

### 4.補. learned 候補の無駄手枝刈り【実装済み・2026-07-12】

C3過大（@19/@102 無駄攻撃）・C2過大（@38 無駄ドン）の一部は**学習不要の探索側で直った**。
learned MCTS の候補生成（`adapter.OPCGGame.legal_actions`）が L1/α-β の枝刈り
（`_prune_futile_attacks`/`_prune_don_moves`）を掛けていなかった＝net が無駄手に visit を割ける
配線の穴。同じ枝刈りを適用（`SERVE_PRUNE_FUTILE`・serve+自己対戦の両方）＝候補集合レベルで
@19/@102/@38 の記録無駄手が消える。検証: arena gen4(枝刈りON) vs gen3 = 0.750（v4採用時 0.583
を上回る・退行なし）・make test 1344 passed。残る @72/@93（届く/意味ある付与だが戦略的に無駄）は
net 判断＝学習側（本走の主眼）。

> 実装順序: 1（R3 serve A/B＝学習不要で先に効果確認）→ 3（encoder＝データ再生成が要るので確定）→
> 2（生成側マークシード）→ 4b/5（学習側）→ 6（ゲート）→ smoke → 本走。
> **二大主眼は R3/C4（負けq飽和）と D1/C2（付与）**（調査で C1「守りすぎ」は単独主眼から外れ、
> その実体は C4 と軽較正に還元された）。
